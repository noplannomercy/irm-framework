# IRM (Intent Routing Map) Framework v10

> AI 에이전트 시스템 설계 시, 고객 요청의 의도를 **행위 유형**으로 분류하고, 성숙도에 따라 적절한 기술로 매핑하는 프레임워크.
> 최종 갱신: 2026-03 v10

---

## 개요

### LLM Native = AX이자 DX

LLM Native 아키텍처는 두 가지를 동시에 실현한다.

- **AX (Agent Experience)**: 에이전트가 업무를 직접 실행한다 — 자연어 한 마디로 전체 절차가 완료된다
- **DX (Digital Transformation)**: 기존 시스템이 에이전트 인프라로 전환된다 — 사람용 UI를 만들면 에이전트용 도구가 자동으로 딸려온다

이 두 가지는 분리된 목표가 아니다. 같은 코드베이스를 두 가지 방식으로 접근하는 것이다.

```
사람용:    웹 UI (바이브코딩으로 생성)
에이전트용: CLI → REST API → IRM 오케스트레이션
```

바이브코딩으로 웹앱을 만들 때 CLI 버전을 함께 만들면, 웹앱 개발 비용은 그대로이고 에이전트 인프라는 덤으로 얻는다.
IRM은 이 에이전트용 인프라의 설계·배치·운영 프레임워크다. IRM이 완성되는 순간 조직의 모든 업무가 자연어로 오케스트레이션된다.

---

### IRM이란?

기존 AI 에이전트 설계는 "데이터가 어디 있느냐"에 따라 기술을 선택했다. 관계DB면 SQL, 문서면 벡터검색. 그러나 같은 데이터라도 **고객의 의도**에 따라 적절한 기술 경로가 달라진다.

예: "매출" 데이터를 다루더라도
- "지금 매출이 얼마야?" → 위치 아는 데이터 조회 (접근)
- "매출이 왜 떨어졌어?" → 과거 문서/이력 탐색 (탐색)
- "매출 보고서 만들어줘" → 데이터 조회 + 정해진 포맷으로 생성 (접근 + 실행)

IRM은 이 분류를 체계화한다.

### 핵심 판단 트리

모든 분류는 **세 가지 질문**으로 결정된다.

```
Q1. 데이터 위치를 아는가?
├── 모른다 (사내 정보) → 탐색(내부)
├── 모른다 (사외 정보) → 탐색(외부)
└── 안다 → 접근 (위치가 확정된 데이터를 가져온다)

Q2. 처리 절차가 있는가?
├── 없다 → 끝
└── 있다 → 실행 (절차를 순서대로 처리한다)

Q3. 이 절차는 결정적이고 재사용 가능한가?
├── Yes → 실행(확정) - 같은 입력이면 같은 결과, 다른 곳에서도 활용 가능
└── No → 실행(유연) - 상황마다 판단 필요, 특정 맥락에 한정
```

### 행위 유형 정의

| 행위 유형 | 설명 | 비유 |
|----------|------|------|
| 탐색(내부) | 사내 어딘가에 있는 정보를 의미 기반으로 찾는다 | 사내 도서관에서 관련 책을 검색하는 것 |
| 탐색(외부) | 조직 밖의 정보를 웹에서 수집한다 | 서점에 가서 새 책을 사오는 것 |
| 접근 | 위치를 아는 데이터에 확정적으로 가져온다 | 119에 전화하는 것 |
| 실행(확정) | 결정적이고 재사용 가능한 절차를 실행한다 | 표준 매뉴얼대로 작업하는 것 |
| 실행(유연) | 상황에 따라 판단하며 절차를 실행한다 | 전문가가 상황 보고 판단하는 것 |

### 행위 유형과 기술의 관계

행위 유형은 **Phase 0에서 분류**한다. 기술은 **Phase 1에서 성숙도에 따라 결정**한다.

| 행위 유형 | 초기 구현 | 표준화 시 |
|----------|---------|---------|
| 탐색(내부) | RAG (pgvector) | → GraphRAG 승격 (pgvector + Apache AGE) |
| 탐색(외부) | 웹 검색 / 크롤링 | 웹 검색 / 크롤링 |
| 접근 | 직접 API/도구 호출 | → CLI + REST API 표준화 |
| 실행(확정) | 프롬프트 (절차 안내) | → CLI 파이프라인형 승격 |
| 실행(유연) | 프롬프트 (판단+실행) | → CLI 에이전트형 승격 |

> **실행 인터페이스 표준: CLI + REST API**
>
> 실행 레이어의 표준 인터페이스는 CLI + REST API다.
>
> ```
> 레거시/신규 시스템
>     ↓ Adapter 패턴 (기존 로직 zero touch)
> CLI (표준화된 입출력 — JSON)
>     ↓ REST 게이트웨이 (FastAPI 등)
> 에이전트/LLM
> ```
>
> - **CLI가 기본 실행 단위**다. 모든 외부 시스템 실행은 CLI로 표준화한다
> - **REST API가 CLI를 감싸는 게이트웨이**다. 네트워크 노출, 인증, 로깅을 담당한다
> - **Adapter 패턴**: 레거시 시스템에 CLI를 덧씌운다. 기존 로직을 건드리지 않고 입출력만 통제한다
> - **권한/보안**: 각 시스템의 기존 권한 체계를 그대로 위임한다. 에이전트 레이어에서 권한을 재구현하지 않는다
>
> 참고: MCP 등 다른 프로토콜로도 노출 가능하다. CLI + REST가 기반이면 위에 어떤 인터페이스든 얹을 수 있다.

> **탐색 표준: RAG → GraphRAG**
>
> 탐색(내부) 표준이 성숙도 기반 두 단계로 분리된다.
>
> ```
> 초기: pgvector 단독 RAG (벡터 유사도 검색)
>   ↓ GraphRAG 승격 조건 충족 시
> 고도화: pgvector + Apache AGE (GraphRAG)
> ```
>
> GraphRAG가 RAG를 대체하는 것이 아니라 **보강**한다. 벡터 검색(pgvector)이 top-k 문서를 뽑고, 그래프(AGE)가 관계 체인으로 컨텍스트를 보강한다.
>
> 탐색 레이어는 단일 인터페이스(REST API)로 노출된다. 호출하는 쪽은 내부 구현(RAG/GraphRAG)을 알 필요 없다.
>
> ```
> POST /search {"query": "..."}
>     → 내부: pgvector + AGE
>     → 반환: 조립된 컨텍스트
> ```
>
> → 별첨: GraphRAG 설계 방법론

> **핵심:** Phase 0에서는 기술명을 쓰지 않는다. "이건 GraphRAG", "이건 CLI 파이프라인형"이 아니라 "이건 탐색", "이건 실행(확정)"으로 분류한다. 어떤 기술로 구현할지는 Phase 1에서 성숙도를 보고 결정한다.

### 체이닝

하나의 요청이 여러 행위 유형에 걸칠 수 있다. 이 경우 체이닝한다.

예: "장애 대응 절차 시작해"
1. 접근 → 모니터링 시스템에서 현재 서버 상태 조회
2. 탐색 → 과거 유사 장애 사례 검색
3. 실행(유연) → 상황에 맞게 판단하며 대응 절차 실행

## Phase 0: 분류

**목적:** 고객의 리소스와 요청 패턴을 파악하여 행위 유형을 매핑한다.

**입력:** 고객의 리소스 (파일, 시스템, 매뉴얼)

**산출물:** IRM 매핑표 (문서 3장)

**수행 주체:** 고객이 직접 / 아키텍트가 인터뷰 / LLM에 위임 — 모두 가능

> **표준 수행 패턴: LLM 초안 → 고객 검증**
>
> 실무에서 가장 효과적인 방식은 다음과 같다:
> 1. LLM에 도메인과 상황만 던져서 Phase 0 시뮬레이션 초안을 뽑는다
> 2. 초안이 80% 채워진 템플릿이 된다
> 3. 고객 미팅에서 "이거 맞아요? 빠진 거 있어요?" 확인하거나, 각 부서에 배포해서 수정·보완하게 한다
> 4. 수집된 결과로 매핑표를 확정한다
>
> 빈 종이에서 시작하는 것보다 초안을 고치는 게 빠르고, 고객 부담도 적다.

> **참고: 성숙도에 따른 자연스러운 진화**
>
> 최초 분석 시 대부분의 절차는 실행(유연)으로, 대부분의 시스템 접근은 직접 호출로 식별되는 것이 정상이다.
>
> 표준화는 운영 과정에서 점진적으로 일어난다:
> 1. 처음에는 직접 호출 + 프롬프트로 시작
> 2. 반복하면서 패턴이 발견됨
> 3. 패턴이 안정화되면 CLI + REST API로 승격
>
> Phase 0 시점에서 CLI 표준화가 거의 없는 것은 자연스러운 상태이다.

### Step 1: 리소스 인벤토리

고객에게 "뭐가 있어요?"를 확인한다. 네 유형으로 나열한다.

| 유형 | 설명 | 예시 |
|------|------|------|
| 파일/문서 | 사내 비정형 데이터. 텍스트, PDF, 위키 등 | Confluence 페이지, 공유폴더 PDF, Wiki |
| 시스템 | 사내 API, DB 등 접근 가능한 시스템 | PostgreSQL, Grafana, JIRA, Slack |
| 절차/매뉴얼 | 정해진 순서가 있는 업무 프로세스 | 장애대응 매뉴얼, 배포 절차서, 온보딩 가이드 |
| 외부 소스 | 사외 정보 채널. 조직 밖에서 가져와야 하는 정보 | 규제 기관 사이트, 업계 뉴스, 경쟁사 공개 정보, 기술 블로그 |

### Step 2: 리소스 성격 분류

나열된 리소스마다 판단 트리를 적용한다.

**Q1. 이건 검색될 지식인가, 접근할 시스템인가?**
- 사내 비정형 문서 (위치를 특정할 수 없는 검색 대상) → 탐색(내부)
- 사외 정보 (조직 밖에서 가져와야 하는 정보) → 탐색(외부)
- 시스템/API/DB (위치가 확정된 접근 대상) → 접근
- 둘 다 해당 → 양쪽 모두 등록

**Q2. 이 안에 실행할 절차가 있는가?**
- 있으면 → Q3으로

**Q3. 이 절차는 결정적이고 재사용 가능한가?**

다음 기준으로 판단한다:

| 판단 기준 | 실행(확정) | 실행(유연) |
|----------|----------|----------|
| 결과 예측성 | 같은 입력 → 같은 결과 | 상황마다 경로/결과 다름 |
| 재사용성 | 다른 프로젝트/팀에서도 활용 가능 | 특정 맥락에 한정 |
| LLM 판단 | 불필요 또는 최소 | 중간중간 판단 필요 |
| 변경 빈도 | 안정적 | 수시 변경 가능 |
| 입력 형태 | 정형적, 예측 가능 | 비정형, 상황마다 다름 |

- 대부분 해당 → 실행(확정)
- 대부분 미해당 → 실행(유연)

> **주의:** 하나의 리소스가 여러 행위 유형에 걸칠 수 있다.
> 예: "장애 대응 매뉴얼"은 탐색(지식으로 검색) + 실행(유연)(절차로 실행) 양쪽에 등록된다.

|---|---------|:--------:|:--------:|:---:|:---------:|:---------:|-------------|

| 4 | 장애 대응 매뉴얼 | O | | | | O | 비결정적: 장애 유형마다 판단 다름, LLM 판단 필요 |

### Step 3: 질의 매핑

분류된 리소스를 기반으로 "이걸 가지고 사람들이 뭘 물어볼까?"를 도출한다. 각 질의에 행위 유형을 태깅한다.

**산출물: 문서 3 — IRM 매핑표**

| # | 예상 질의 | Q1 위치 | Q2 절차 | Q3 결정적/재사용 | → 행위 유형 | 관련 리소스 |
|---|----------|:-------:|:------:|:--------------:|-----------|-----------|
| 1 | 이 에러코드 뭐야? | X | X | - | 탐색 | #1 포스트모템 |
| 2 | 서버 CPU 알려줘 | O | X | - | 접근 | #6 모니터링 |
| 3 | 보안 정책 뭐야? | X | X | - | 탐색 | #3 보안정책 |
| 4 | 작년 유사 장애 있었어? | X | X | - | 탐색 | #1 포스트모템 |
| 5 | 마지막 배포 언제야? | O | X | - | 접근 | #10 배포시스템 |
| 6 | DB 백업 상태 확인해줘 | O | X | - | 접근 | #7 운영DB |
| 7 | 디스크 용량 확인해줘 | O | X | - | 접근 | #6 모니터링 |
| 8 | 누가 서버 접근권한 있어? | O | X | - | 접근 | LDAP |
| 9 | 점검 일정 알려줘 | O | X | - | 접근 | 캘린더 |

## Phase 1: 배치

Phase 0의 IRM 매핑표(행위 유형)를 기반으로, **성숙도에 따라 기술을 선택**하고, **배치 대상과 함께 각 대상의 최소 설계 요구사항**을 산출한다.

> **Phase 1의 원칙:**
> Phase 1은 "뭘 만드느냐"와 "어떤 요구사항으로 만드느냐"까지 정의한다. "어떻게 만드느냐"는 Phase 2의 영역이다.
> 건축으로 치면 "화장실 2개, 방 3개, 방에는 창문 필수"가 Phase 1이고, "어떤 자재로 어떻게 짓느냐"가 Phase 2이다.
>
> 요구사항은 Phase 0 Step 3의 질의 매핑표를 역추적하여 도출한다. 질의 패턴에서 "이 데이터를 어떤 기준으로 찾아야 하는가", "이 시스템에 어떤 동작이 필요한가"가 자연스럽게 드러난다. Phase 0을 반복할수록 질의가 늘고, 질의가 늘수록 요구사항이 정교해진다.

### 1-A: 탐색

"탐색"으로 분류된 리소스를 정리한다. 탐색은 내부와 외부로 나뉘며, 기술이 다르다.

| 구분 | 대상 | 초기 기술 | 고도화 기술 |
|------|------|---------|-----------|
| 탐색(내부) | 사내 문서, 위키, 이력 등 | RAG (pgvector) | GraphRAG (pgvector + Apache AGE) |
| 탐색(외부) | 규제 동향, 업계 뉴스, 경쟁사 정보 등 | 웹 검색 / 크롤링 | 웹 검색 / 크롤링 |

> **탐색(내부) GraphRAG 승격 기준**
>
> 다음 조건 중 2개 이상 충족 시 GraphRAG 승격을 검토한다:
> - 도메인 내 동음이의어/중의어가 많다 (예: 한도, 금리, 처리, 승인 등)
> - 관계/인과 기반 질의 패턴이 발견된다 ("A가 B에 영향주나?", "A 때문에 B가 발생하나?")
> - 순수 RAG 히트율이 기대 수준 이하다
> - 문서 간 관계 추적이 필요하다 (동일 엔티티가 여러 문서에 다른 맥락으로 등장)
>
> **GraphRAG 인터페이스 원칙**
>
> 탐색 레이어는 단일 REST API 엔드포인트로 노출한다. 호출하는 쪽은 내부 구현(RAG/GraphRAG)을 알 필요 없다.
> ```
> POST /search {"query": "..."}  →  {"context": "..."}
> ```
> 초기(pgvector 단독)와 고도화(GraphRAG) 모두 동일 인터페이스를 유지한다.

## 성숙도 승격 기준

### 탐색(내부): RAG → GraphRAG 승격 체크리스트

- [ ] 동음이의어/중의어로 인한 검색 오류가 반복 발생한다
- [ ] 관계/인과 기반 질의 패턴이 전체 질의의 20% 이상이다
- [ ] RAG 히트율이 목표치 대비 지속 미달이다
- [ ] 동일 엔티티가 여러 도메인에 다른 의미로 등장한다
- [ ] pgvector + Apache AGE 인프라 구성 완료 (Docker 또는 관리형 PostgreSQL)

**승격 시 추가 작업:**
- Ingest 파이프라인에 엔티티+트리플 추출 단계 추가 (LLM 기반 자동 추출)
- 그래프 공간(graph space) 생성 및 기존 문서 재처리
- search 엔드포인트 내부 로직 교체 (외부 인터페이스 불변)

### 접근: 직접 호출 → CLI + REST API 승격 체크리스트

- [ ] 동일 시스템에 대한 접근 패턴이 3가지 이상 반복된다
- [ ] 여러 에이전트/환경에서 동일 시스템에 접근이 필요하다
- [ ] 접근 추적/감사 로그가 필요하다
- [ ] 레거시 시스템 Adapter CLI 작성 완료
- [ ] REST 게이트웨이 배포 완료

### 실행(확정): 프롬프트 → CLI 파이프라인형 승격 체크리스트

- [ ] 동일 절차가 월 10회 이상 실행된다
- [ ] 외부 시스템 실행이 포함된다
- [ ] 절차 실행 추적/재현이 필요하다
- [ ] 단계별 성공/실패 분리가 필요하다
- [ ] CLI 단위 테스트 작성 완료

### 실행(유연): 프롬프트 → CLI 에이전트형 승격 체크리스트

- [ ] 외부 시스템 실행 추적이 필요해졌다
- [ ] 동일 판단 패턴이 3회 이상 반복 확인됐다
- [ ] 판단(LLM) / 실행(CLI) 분리가 명확히 가능하다

### 강등 기준

승격 후에도 다음 조건이 지속되면 이전 단계로 강등한다:
- GraphRAG: 그래프 관계 히트율이 벡터 단독 대비 개선 없음 → RAG 단독 유지
- CLI 파이프라인형: 실행 빈도 월 3회 미만으로 감소 → 프롬프트로 강등
- CLI 에이전트형: 판단 패턴이 고정되지 않고 계속 변화 → 프롬프트 유지

## 운영: IRM 매핑표 유지보수

IRM 매핑표는 한 번 만들고 끝이 아니다. 다음 경우 갱신한다:
- 새 시스템/문서가 추가될 때
- 기존 절차가 변경될 때
- 예상 못한 질의 패턴이 발견될 때
- 승격 체크리스트 조건 충족 시 단계 갱신
- 강등 기준 해당 시 단계 갱신
