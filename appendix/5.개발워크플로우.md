## 2. 신규 개발

**gstack + superpowers 풀 코스.**

```
Phase 0 요구사항:
  SRS 작성 (요구사항 확정) → 환경 구성

Phase 1 설계:
  /office-hours (SRS 기반)
  → superpowers:brainstorming
  → /plan-ceo-review (선택, HOLD SCOPE 모드)
  → superpowers:writing-plans
  → /plan-eng-review
  → /plan-design-review

Phase 2 구현:
  feature 브랜치 생성
  → superpowers:executing-plans (서브에이전트 병렬 가능)
  → superpowers:test-driven-development
  (버그 시) superpowers:systematic-debugging

Phase 3 검증+출하:
  superpowers:verification-before-completion
  → /review → /qa → /design-review (선택)
  → /ship → /retro
```

### 주의사항
- SRS가 /office-hours보다 먼저 — 요구사항 없이 돌리면 방향 틀어짐
- /plan-ceo-review는 별에별 기능 다 붙음 — HOLD SCOPE 필수, MVP면 생략
- /plan-eng-review는 진짜 문제를 잡아줌 — 가치 높음
- superpowers만보다 gstack 조합이 훨씬 빠름 — 검증+출하 자동화가 핵심

## 3. 기능 추가

**신규 개발의 경량 버전.**

```
Phase 0 요구사항:
  기능 스펙 정의 (간단한 SRS 또는 한 줄 설명)

Phase 1 설계:
  superpowers:brainstorming (요구사항 기반)
  → superpowers:writing-plans
  → /plan-eng-review (기존 코드 충돌 확인)

Phase 2 구현:
  feature 브랜치 생성
  → superpowers:executing-plans
  → superpowers:test-driven-development

Phase 3 검증+출하:
  superpowers:verification-before-completion
  → /review → /qa → /ship
```

### 신규 개발 대비 생략 항목

| 생략 | 이유 |
|------|------|
| SRS (풀) | 간단한 스펙이면 충분 |
| /office-hours | 제품 방향 이미 잡혀있음 |
| /plan-ceo-review | 기존 제품에 기능 얹는 거라 과함 |
| /plan-design-review | UI 변경 클 때만 |
| /design-review | 필요시만 |
| /retro | 필요시만 |

### 체감
- 테스트 환경 이미 있으면 /qa + /ship 2분 컷
- 두 번째부터는 셋업 없이 순수 테스트+출하만

## 주의사항 모음

- **요구사항 먼저** — brainstorming이든 /office-hours든 요구사항 없이 시작하면 방향 틀어짐
- **/plan-ceo-review 주의** — 원론적 문제 파악 컨셉이라 별에별 기능 다 붙음. HOLD SCOPE 모드 필수
- **/plan-eng-review 가치 높음** — 진짜 기술 문제를 잡아줌. 단 얕을 수 있어 "이게 다야?" 적극 질문
- **/qa 명시적 지시** — "수정까지 해줘" 안 하면 /qa-only로 빠질 수 있음
- **gstack 표면적 이슈** — "한 이슈 = 한 질문" 강제라 깊은 문제 놓칠 위험
- **두 번째부터 빠름** — 테스트 환경 한 번 깔리면 이후 /qa + /ship은 2분 컷
