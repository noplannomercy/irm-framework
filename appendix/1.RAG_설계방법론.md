> 입력: Phase 1 1-A-1 (데이터 소스 목록) + 1-A-2 (탐색 요구사항)

## §1. 컬렉션 설계

### 목적
데이터 소스를 도메인 경계에 따라 컬렉션으로 분리하고, 벡터 저장소의 제품과 인덱스 타입을 결정한다. 컬렉션 분리가 전체 RAG 설계의 기반이 되며, 이후 §2~§8의 모든 의사결정은 컬렉션 단위로 수행된다.

### 프레임워크 용어: "컬렉션"

IRM에서 "컬렉션"은 벡터 저장소 내의 논리적 데이터 격리 단위를 의미한다. 구현 시 각 제품의 대응 개념으로 매핑한다.

| 프레임워크 용어 | Qdrant | pgvector | Milvus | OpenSearch | Pinecone | Chroma |
|-------------|--------|----------|--------|-----------|----------|--------|
| **컬렉션** | Collection | Table | Collection | Index | Namespace | Collection |

### 단일 컬렉션 vs 다중 컬렉션 판단 기준

| 판단 기준 | 단일 | 다중 |
|----------|------|------|
| 데이터 갱신 주기가 유사 | O | |
| 갱신 주기가 크게 다름 (정적 정책 vs 실시간 로그) | | O |
| 메타데이터 스키마가 통일 가능 | O | |
| **도메인 고유 메타데이터가 다름** | | **O** |
| 접근 권한 분리가 필요 | | O |
| 검색 시 질의 패턴이 다름 | | O |
| 관리 단순성 우선 | O | |

### 컬렉션 분리 기준

1-A-1의 데이터 소스 목록과 1-A-2의 메타데이터 후보를 보고, 다음 기준으로 분리한다:

1. **도메인 메타데이터가 다른가?** — 인시던트 이력에는 `severity`가 필요하고 정책 문서에는 불필요하다면, 같은 컬렉션에 넣으면 스키마가 낭비된다.
2. **갱신 주기가 크게 다른가?** — 정적 정책 문서와 수시로 쌓이는 인시던트 이력은 갱신 파이프라인이 다르다.
3. **검색 시 함께 조회되는가?** — "정책 기준 알려줘"와 "인시던트 이력 찾아줘"는 거의 섞이지 않는다. 분리하면 검색 범위가 좁아져 정밀도가 올라간다.

### 컬렉션 구조 설계 예시

```
벡터 저장소
├── 컬렉션: policy_and_guidelines
│   └── 포함: 정책 문서, 부서별 가이드라인, 민감도 분류 기준, 규정, 절차 문서(지식)
│
├── 컬렉션: vendor_and_tools
│   └── 포함: 벤더 계약서, SLA, 도구 승인 목록, 보안 감사 리포트
│
└── 컬렉션: incidents
    └── 포함: 인시던트 이력(JIRA 티켓)
```

### 벡터 저장소 제품 비교

| DB | 타입 | 강점 | 적합한 규모 |
|----|------|------|-----------|
| pgvector | PostgreSQL 확장 | 기존 인프라 활용, 관리 단순, 테이블=컬렉션으로 자연스러운 분리 | 소~중규모 |
| Qdrant | 전용 벡터DB | 필터링 성능, 하이브리드 검색 내장, 스칼라 양자화 | 중~대규모 |
| Milvus | 전용 벡터DB | 대규모 분산, GPU 지원 | 대규모 |
| OpenSearch | 검색 엔진 + 벡터 | BM25+벡터 하이브리드 네이티브, Docling 공식 연동 | 중~대규모 |
| Chroma | 경량 벡터DB | 프로토타이핑, 로컬 개발 | 소규모/PoC |
| Pinecone | 매니지드 서비스 | 운영 부담 최소, 서버리스 | 모든 규모 (비용 고려) |

### 인덱스 타입 선택

| 인덱스 | 장점 | 단점 | 적합 |
|--------|------|------|------|
| Flat (정확 검색) | 정확도 100% | 느림 | 10만 건 이하 |
| HNSW | 빠르고 정확 | 메모리 사용 높음 | 범용 (가장 추천) |
| IVF | 메모리 효율적 | 정확도 약간 하락 | 100만 건 이상 |

## §2. 컬렉션별 메타데이터 스키마 확정

### 목적
§1에서 분리한 각 컬렉션에 대해, 1-A-2의 메타데이터 후보를 기반으로 **공통 메타데이터**와 **도메인 메타데이터**를 설계한다. 컬렉션마다 스키마가 다르다.

### 메타데이터 구조: 공통 + 도메인

| 구분 | 설명 | 모든 컬렉션 동일 |
|------|------|:---------------:|
| **공통 메타데이터** | 전 컬렉션에 공통으로 존재하는 필드 | O |
| **도메인 메타데이터** | 해당 컬렉션에서만 의미 있는 필드 | X (컬렉션마다 다름) |

### 의사결정 순서

**1단계: 1-A-2에서 후보 수집**

1-A-2 표의 "메타데이터 후보" 컬럼을 전부 모은다.

**2단계: 공통/도메인 분류**

| 분류 기준 | → 등급 |
|----------|--------|
| 2개 이상 컬렉션에서 사용됨 | 공통 |
| 1개 컬렉션에서만 사용됨 | 해당 컬렉션의 도메인 메타 |
| 질의에서 직접 사용되지 않음 | 보류 |

**3단계: 추출 방식 결정**

| 추출 방식 | 적합한 경우 | 예시 |
|----------|-----------|------|
| 자동 추출 (LLM 기반) | 비정형 텍스트에서 구조화 필요 | 인시던트 유형, 심각도, 원인 분류 |
| 자동 추출 (규칙 기반) | 패턴이 명확한 경우 | 날짜, 문서 버전, 부서명 |
| 수동 태깅 | 자동 추출 정확도가 낮은 경우 | 데이터 민감도 등급 |
| 소스 시스템에서 가져오기 | 이미 구조화되어 있는 경우 | JIRA 티켓의 우선순위, 상태 |

**4단계: 컬렉션별 스키마 정의서 작성**

컬렉션마다 별도로 정의한다.

### 스키마 설계 예시

**공통 메타데이터 (전 컬렉션):**

```json
{
  "doc_type": "string",    // 문서 유형
  "dept_name": "string",   // 관련 부서 ("all" = 전사)
  "date": "date",          // 작성일/발생일
  "source": "string"       // 원본 위치 (URL, ID)
}
```

**컬렉션: policy_and_guidelines 도메인 메타:**

```json
{
  "scope": "string",              // 적용 범위 (company-wide / dept-specific)
  "data_sensitivity": "string",   // 민감도 등급 (수동 태깅)
  "regulation_type": "string"     // 관련 규제 (eu_ai_act / gdpr / pipa 등)
}
```

**컬렉션: vendor_and_tools 도메인 메타:**

```json
{
  "tool_name": "string[]",        // LLM 도구명 목록
  "approval_status": "string",    // approved / pending / rejected / deprecated
  "vendor_type": "string"         // openai / anthropic / google / other
}
```

**컬렉션: incidents 도메인 메타:**

```json
{
  "incident_type": "string",      // data_leak / misuse / malfunction / other
  "severity": "string",           // critical / high / medium / low
  "tool_name": "string[]",        // 관련 도구명
  "root_cause": "string"          // 원인 분류 (LLM 추출)
}
```

### 주의사항
- 공통 메타데이터는 3~5개가 적정. 도메인 메타데이터는 컬렉션당 2~4개.
- `tool_name`이 vendor_and_tools와 incidents 양쪽에 있는 건 의도적. 도구 기준 교차 검색이 필요하기 때문.
- Phase 0 반복 시 질의가 추가되면 도메인 메타가 늘어날 수 있다. 컬렉션 단위로 확장.
- 자동 추출 시 정확도를 반드시 샘플링 검증. 80% 미만이면 수동 태깅 또는 규칙 보완.

## §3. 문서 파싱 및 청킹

### 목적
문서를 벡터 저장소에 넣기 위해 파싱하고 적절한 단위로 분할한다. 2026년 현재, 청킹 전략의 핵심은 "어떻게 자를까?"가 아니라 **"어떤 도구로 파싱+청킹을 처리할까?"** 이다.

### 현대적 접근: 도구 기반 파싱+청킹

기존 방식(고정 사이즈, 시맨틱 분할 등)은 문서 구조를 무시하고 텍스트만 처리했다. Docling 같은 도구는 문서의 레이아웃, 헤더 계층, 테이블, 이미지를 인식한 후 구조 기반으로 청킹한다.

### 문서 파싱+청킹 도구 비교

| 도구 | 접근 방식 | 강점 | 적합 |
|------|---------|------|------|
| **Docling + HybridChunker** | 문서 구조 인식 → 계층적 청킹 → 토크나이저 인식 정제 | 테이블/이미지 처리, 임베딩 모델 토크나이저 연동, LF AI 재단 프로젝트 | PDF, DOCX, PPTX 등 복합 문서 (**권장**) |
| Unstructured | 파티셔닝 기반 추출 | 다양한 포맷 지원, LangChain/LlamaIndex 연동 | 범용 파이프라인 |
| LlamaIndex SentenceSplitter | 문장 기반 분할 | 구현 단순, 프레임워크 내장 | 텍스트 위주 문서 |
| 재귀적 분할 (RecursiveCharacterTextSplitter) | 구분자 기반 재귀 분할 | 마크다운/위키에 적합 | 이미 구조화된 텍스트 |

### Docling + HybridChunker 사용 가이드 (권장)

HybridChunker는 두 단계로 동작한다:
1. **HierarchicalChunker**: 문서 구조(섹션, 단락, 테이블, 그림)를 기반으로 1차 분할
2. **토크나이저 인식 정제**: 너무 큰 청크는 분할하고, 같은 섹션의 작은 청크는 자동 병합 (merge_peers)

**핵심: §4에서 선정한 임베딩 모델의 토크나이저와 반드시 동일한 토크나이저를 사용해야 한다.**

```python
from docling.document_converter import DocumentConverter
from docling.chunking import HybridChunker
from docling_core.transforms.chunker.tokenizer.huggingface import HuggingFaceTokenizer
from transformers import AutoTokenizer

# 1. 문서 파싱
doc = DocumentConverter().convert(source="document.pdf").document

# 2. §4에서 선정한 임베딩 모델과 동일한 토크나이저로 청킹
EMBED_MODEL_ID = "dragonkue/BGE-m3-ko"  # §4에서 결정된 모델
tokenizer = HuggingFaceTokenizer(
    tokenizer=AutoTokenizer.from_pretrained(EMBED_MODEL_ID),
)

chunker = HybridChunker(
    tokenizer=tokenizer,
    merge_peers=True,
)
chunks = list(chunker.chunk(dl_doc=doc))

# 3. 컨텍스트 강화된 텍스트 사용
for chunk in chunks:
    text_to_embed = chunker.contextualize(chunk)  # 상위 헤딩 등 맥락 포함
```

### 컬렉션별 파싱+청킹 전략

§1에서 분리한 컬렉션 단위로 전략을 결정한다.

| 컬렉션 | 데이터 형태 | 도구 | 전략 |
|-------|-----------|------|------|
| policy_and_guidelines | Confluence(HTML), Wiki | Docling (HTML 지원) | 섹션/조항 단위 구조 인식 청킹. merge_peers=True |
| vendor_and_tools | PDF (계약서), Wiki (승인 목록) | **Docling + HybridChunker** (PDF), 재귀적 분할 (Wiki) | PDF는 테이블 구조 인식 필수. 승인 목록은 1 도구 = 1 청크 |
| incidents | JIRA 티켓 | **커스텀 전처리** | 1 티켓 = 1 청크. 메타데이터는 JIRA 필드에서 직접 추출 |

### 테이블 처리 (vendor_and_tools 컬렉션)

벤더 계약서에 비용 조건·SLA 수치 테이블이 포함된다. 마크다운 형태 직렬화를 적용한다.

```python
from docling_core.transforms.chunker.hierarchical_chunker import (
    ChunkingDocSerializer, ChunkingSerializerProvider,
)
from docling_core.transforms.serializer.markdown import MarkdownTableSerializer

class MDTableProvider(ChunkingSerializerProvider):
    def get_serializer(self, doc):
        return ChunkingDocSerializer(
            doc=doc,
            table_serializer=MarkdownTableSerializer(),
        )

chunker = HybridChunker(
    tokenizer=tokenizer,
    merge_peers=True,
    serializer_provider=MDTableProvider(),
)
```

### 주의사항
- 시맨틱 청킹은 비용 대비 효과가 불확실하다는 2025-2026 연구 결과가 있다. 도구 기반 구조 인식 청킹이 가장 안정적.
- 오버랩은 서술형 문서(policy_and_guidelines)에서만 의미 있고, 구조화된 데이터(incidents)에서는 불필요.
- 청크 사이즈가 임베딩 모델 컨텍스트 길이를 초과하면 잘림 발생. HybridChunker + 토크나이저 연동으로 자동 방지.

## §4. 임베딩 모델 선정

### 목적
문서 청크와 질의를 벡터로 변환할 모델을 선택한다. §3의 청킹 파이프라인에 토크나이저를 제공하므로, §3과 §4는 사실상 동시에 결정해야 한다.

### 선정 기준

| 기준 | 검토 항목 |
|------|---------|
| 성능 | MTEB 벤치마크, 도메인별 성능 (네이티브 한국어 데이터 기준) |
| 비용 | API 호출 단가 vs 로컬 추론 비용 |
| 한국어 | 한국어 지원 품질 — 번역 벤치마크가 아닌 네이티브 데이터 기준 |
| 벡터 차원 | 차원 수에 따른 저장 비용/검색 성능 트레이드오프 |
| 컨텍스트 길이 | 청크 사이즈 대비 모델 최대 입력 길이 |

### 주요 모델 비교 (2025-2026 기준)

**한국어 특화/강점 모델:**

| 모델 | 차원 | 컨텍스트 | 한국어 성능 | 비용 | 비고 |
|------|:----:|:-------:|:---------:|------|------|
| KURE-v1 (고려대 NLP&AI) | 1024 | - | 최상위 | 로컬 무료 | 한국어 특화 개발 |
| dragonkue/BGE-m3-ko | 1024 | 8K | 우수 (F1 0.7456) | 로컬 무료 | BGE-M3 한국어 파인튜닝. Dense+Sparse+ColBERT 지원 |
| Upstage embedding | 4096 | 4K | 최우수 | API | 한국어 특화, 높은 차원 |

**다국어 범용 모델:**

| 모델 | 차원 | 컨텍스트 | 한국어 성능 | 비용 | 비고 |
|------|:----:|:-------:|:---------:|------|------|
| BGE-M3 (BAAI) | 1024 | 8K | 우수 | 로컬 무료 | 다국어 최강. Dense+Sparse+ColBERT 3가지 검색 지원 |
| intfloat/multilingual-e5-large-instruct | 1024 | 512 | 우수 | 로컬 무료 | MTEB 한국어 리더보드 상위 |
| OpenAI text-embedding-3-large | 3072 | 8K | 양호 | API | 가장 편리, 범용 |
| OpenAI text-embedding-3-small | 1536 | 8K | 양호 | API (저가) | 비용 대비 성능 양호 |
| Cohere embed-v3 | 1024 | 512 | 양호 | API | 100+ 다국어, 검색 특화 |
| jina-embeddings-v3 | 1024 | 8K | 양호 | API/로컬 | 570M 파라미터, 다목적 |

### 한국어 도메인 선택 가이드

| 상황 | → 추천 모델 |
|------|-----------|
| 한국어 전용, 최고 성능 필요 | KURE-v1 또는 Upstage |
| 한국어 중심 + 로컬 운영 | BGE-m3-ko 또는 BGE-M3 |
| 한국어 + 영어 혼합 | BGE-M3 |
| 빠른 시작, API 기반 | Upstage (한국어) 또는 OpenAI (범용) |
| 데이터 외부 전송 불가 (보안) | KURE-v1 또는 BGE-m3-ko (로컬 필수) |
| 프로토타입 / PoC | OpenAI text-embedding-3-small |

> **중요 경고:** 범용 다국어 모델이 번역된 벤치마크에서는 높은 점수를 보이지만, 실제 한국어 원본 데이터에서는 상당한 성능 격차를 보일 수 있다. 네이티브 한국어 데이터로 반드시 검증할 것.

### API vs 로컬 판단 기준

| 상황 | → 선택 |
|------|--------|
| 초기 프로토타입, 문서 수 적음 | API (빠른 시작) |
| 대규모 인덱싱, 비용 민감 | 로컬 (GPU 필요) |
| 데이터 외부 전송 불가 (보안) | 로컬 (필수) |
| 한국어 최적 성능 | 로컬 (KURE-v1, BGE-m3-ko) |

### 주의사항
- **임베딩 모델을 변경하면 전체 재인덱싱이 필요.** 초기 선정이 매우 중요.
- §3에서 HybridChunker 사용 시, 여기서 선정한 모델의 토크나이저를 반드시 전달.
- 동일 모델로 문서 임베딩과 질의 임베딩을 수행해야 함.

## §5. 검색 전략 결정

### 목적
사용자 질의에 대해 가장 관련성 높은 청크를 가져오는 검색 방식을 결정한다. 검색은 컬렉션 단위로 수행되며, 복합 질의는 여러 컬렉션을 교차 검색할 수 있다.

### 검색 방식 비교

| 방식 | 장점 | 단점 | 적합 |
|------|------|------|------|
| 벡터 검색 단독 | 의미 기반 유사도 | 키워드 매칭 약함 | 서술형 질의 중심 |
| 키워드 검색 (BM25) 단독 | 정확한 키워드 매칭 | 의미적 유사성 못 잡음 | 코드명/고유명사 검색 |
| **하이브리드 (벡터 + BM25)** | 양쪽 장점 결합 | 구현 복잡도 증가 | **대부분의 경우 권장** |

> **2026년 기준 하이브리드 검색이 사실상 표준.** BGE-M3는 Dense+Sparse+ColBERT를 단일 모델에서 지원하며, 주요 벡터DB도 하이브리드 검색을 네이티브 지원.

### 하이브리드 검색 결합 방식

| 방식 | 설명 | 장점 |
|------|------|------|
| RRF (Reciprocal Rank Fusion) | 각 검색 결과의 순위를 역수로 합산 | 가중치 튜닝 불필요, 안정적 |
| 가중치 합산 | 벡터 점수 × α + BM25 점수 × (1-α) | 도메인별 미세 조정 가능 |

> 초기에는 RRF로 시작하고, 검색 결과를 모니터링하며 가중치 합산으로 전환 검토를 권장.

### top-k 설정

| 규모 | top-k 권장 | 근거 |
|------|:---------:|------|
| 소규모 (1만 건 이하) | 3~5 | 청크 수가 적으므로 적은 수로 충분 |
| 중규모 (1~10만 건) | 5~10 | 관련 문서가 분산되어 있을 가능성 |
| 대규모 (10만 건 이상) | 10~20 (리랭킹 전), 3~5 (리랭킹 후) | 리랭킹과 조합 시 효과적 |

### 교차 컬렉션 검색 (§1 연동)

복합 질의가 여러 컬렉션에 걸치는 경우:

```
예: "작년 인시던트 관련 정책 위반 사항 정리해줘"
→ incidents 컬렉션: date >= 2024-01-01인 인시던트 검색
→ policy_and_guidelines 컬렉션: 관련 정책 조항 검색
→ 두 결과를 합쳐서 리랭킹
```

라우팅 프롬프트에서 질의를 분석하여 대상 컬렉션을 결정한다.

### 메타데이터 필터링 적용

§2에서 정의한 컬렉션별 메타데이터를 활용한다.

```
예시:
질의: "마케팅 팀 LLM 사용 가이드라인"
→ 컬렉션: policy_and_guidelines
→ 필터: dept_name = "marketing"

질의: "데이터 유출 인시던트"
→ 컬렉션: incidents
→ 필터: incident_type = "data_leak"

질의: "ChatGPT 승인 상태"
→ 컬렉션: vendor_and_tools
→ 필터: tool_name contains "ChatGPT"
```

> 필터는 질의에서 명확한 조건이 식별될 때만 적용. 모호한 질의에 강제 필터링하면 관련 문서가 누락될 수 있다.

## §6. 리랭킹 적용 여부

### 목적
초기 검색 결과의 정밀도를 높이기 위해 리랭킹 모델 적용 여부를 결정한다.

### 적용 판단 기준

| 상황 | → 결정 |
|------|--------|
| 검색 결과에서 관련 없는 청크가 자주 섞임 | 적용 |
| top-k를 높게 잡아야 원하는 결과가 나옴 | 적용 |
| 하이브리드 검색 사용 중 | 적용 권장 |
| 교차 컬렉션 검색 결과를 통합 정렬해야 함 | 적용 권장 |
| 문서가 소규모이고 검색 품질 충분 | 미적용 |
| 응답 지연에 민감한 실시간 시스템 | 미적용 또는 경량 모델 |

### 리랭킹 모델 비교 (2025-2026 기준)

**API 기반:**

| 모델 | 다국어 | 속도 | 정밀도 | 비고 |
|------|:-----:|:----:|:-----:|------|
| ZeRank 2 (ZeroEntropy) | O (100+) | ~60ms | 최상위 | 2026 리더보드 1위, 교정된 점수 |
| Cohere Rerank 3.5 | O (100+) | ~100ms | 높음 | 안정적, Nimble 버전 |
| Voyage Rerank 2.5 | O | ~600ms | 매우 높음 | 금융/법률 정밀도 최우선 |
| Jina Reranker v3 | O (18개 언어) | 빠름 | SOTA (61.94 nDCG-10) | Qwen3-0.6B 기반, 최신 |

**오픈소스 (로컬):**

| 모델 | 다국어 | 속도 (GPU) | 정밀도 | 비고 |
|------|:-----:|:---------:|:-----:|------|
| BGE-reranker-v2-m3 | O (100+) | ~80ms | 높음 | Apache 2.0, **로컬 최적** |
| FlashRank | O | 매우 빠름 | 양호 | CPU 동작, 프로토타이핑용 |
| mxbai-rerank-v2 | O | 빠름 | 높음 | 구조화 데이터 강점 |

### 한국어 환경 리랭킹 추천

| 상황 | → 추천 |
|------|--------|
| 한국어 + 비용 민감 + GPU 보유 | BGE-reranker-v2-m3 (로컬) |
| 한국어 + API 선호 | ZeRank 2 또는 Cohere 3.5 |
| 한국어 + 최고 성능 | Jina Reranker v3 |
| 프로토타입 / CPU 환경 | FlashRank |

### 리랭킹 파이프라인

```
하이브리드 검색 (컬렉션별 또는 교차)
→ top-k (20~50건) 후보 추출
→ 리랭킹
→ 최종 top-n (3~5건) → LLM 전달

주의:
- 교차 컬렉션 검색 시, 각 컬렉션 결과를 합친 후 리랭킹 1회 수행.
  컬렉션별로 따로 리랭킹하면 컬렉션 간 상대 순위를 비교할 수 없다.
- ZeRank 2는 교정된 확률 점수 출력 → score > 0.7 임계값 필터 가능.
  BGE는 raw score → 쿼리마다 점수 범위가 달라 임계값 튜닝 필요.
```

## §7. 인덱싱 범위 결정

### 목적
검색 결과의 정확성을 보장하기 위해 각 컬렉션의 인덱싱 범위를 결정한다. 핵심 질문은 **"이 질의에 어떤 데이터가 히트나야 올바른 결과인가?"** 이다. 보관이 아니라 검색 결과 품질이 기준이다.

### 의사결정 순서

**1단계: Phase 0 Step 3 질의 매핑표 역추적**

Step 3의 각 질의에서 "올바른 검색 결과"가 무엇인지를 역으로 정의한다.

```
예:
질의: "작년 유사 인시던트에서 어떻게 대응했어?"
→ 올바른 결과: 작년 이전 인시던트도 히트나야 함 (패턴 분석용)
→ 이 컬렉션은 이력형

질의: "배포 절차가 어떻게 돼?"
→ 올바른 결과: 최신 버전 런북만 히트나야 함 (구버전 절차로 배포하면 위험)
→ 이 컬렉션은 최신형
```

**2단계: 컬렉션별 이력형/최신형 결정**

§1에서 분리한 각 컬렉션이 이력형인지 최신형인지 먼저 결정한다. 이 결정이 인덱싱 범위의 근간이다.

| 유형 | 정의 | 검색 결과 기대 | 인덱싱 전략 |
|------|------|-------------|-----------|
| **이력형** | 오래된 데이터도 검색 결과에 필요 | 과거 사례, 패턴, 추이를 포함한 결과 | 기간 범위를 넓게. 조건부로 제한 |
| **최신형** | 최신 버전만 히트나야 함 | 현재 유효한 정책, 절차, 기준만 반환 | 최신 버전만. 구버전 히트는 **오류** |

**3단계: 유형별 세부 기준 결정**

이력형과 최신형에 따라 기간, 제외 조건 등 세부 사항을 결정한다.

### 이력형 컬렉션 인덱싱 기준

이력형은 "어디까지 과거를 포함할 것인가"가 핵심이다.

| 기준 | 판단 방법 | 예시 |
|------|---------|------|
| 질의의 시간 범위 | Step 3에서 "작년", "지난 분기" 등 시간 표현 확인 | "작년 유사 인시던트" → 최소 2년 |
| 패턴 분석 필요 여부 | 반복 패턴 분석에 몇 년치가 필요한가 | 분기별 비용 추이 → 최소 3년 |
| 상태 기반 제한 | 진행 중 건은 접근(실시간 조회)으로 처리 | 종결 건만 인덱싱 |
| 심각도 기반 우선 | 중요 건은 기간 무관 유지 | severity=critical → 기간 무관 전수 |

### 최신형 컬렉션 인덱싱 기준

최신형은 **"구버전이 히트나면 안 된다"** 가 핵심이다.

| 기준 | 판단 방법 |
|------|---------|
| 최신 버전만 유지 | 문서 수정 시 구버전 청크를 반드시 삭제 후 신버전 삽입 |
| 구버전 히트 위험도 | 구버전 정책/절차로 행동하면 어떤 문제가 생기는가 (위험할수록 삭제 철저) |
| 드래프트 제외 | 미승인 문서가 히트나면 잘못된 정보를 근거로 판단할 위험 |

### 인덱싱 제외 대상 (검색 결과 오염 방지 관점)

| 제외 대상 | 제외 이유 (검색 결과 관점) |
|---------|----------------------|
| 구버전 문서 | 최신형 컬렉션에서 구버전이 히트나면 잘못된 판단 유도 |
| 드래프트/미승인 문서 | 확정 안 된 내용이 검색 결과에 포함되면 신뢰도 하락 |
| 중복 문서 | 동일 내용이 여러 청크로 히트나면 LLM 컨텍스트 낭비 |
| 정보 가치 없는 자동 생성 항목 | 노이즈가 검색 결과에 섞이면 정밀도 하락 |

## §8. 갱신 파이프라인 설계

### 목적
검색 결과가 outdated 되지 않도록 벡터 저장소를 최신 상태로 유지한다. 핵심 질문은 **"이 데이터가 갱신 안 됐을 때 검색 결과에 어떤 문제가 생기는가?"** 이다.

### 의사결정 순서

**1단계: 갱신 지연의 검색 결과 영향도 평가**

각 컬렉션에서 "데이터가 갱신 안 됐을 때" 어떤 문제가 생기는지를 먼저 평가한다. 이것이 갱신 긴급도를 결정한다.

| 컬렉션 유형 (§7 기준) | 갱신 안 됐을 때 검색 결과 문제 | → 갱신 긴급도 |
|---------------------|--------------------------|:----------:|
| **최신형** (정책, 절차, 기준) | 구버전 정책이 히트 → 잘못된 판단 근거 제공 → **위험** | **즉시** |
| **최신형** (도구 승인 목록) | deprecated 도구가 "승인됨"으로 히트 → 잘못된 안내 | **즉시** |
| **이력형** (인시던트 이력) | 최신 인시던트가 누락 → 유사 사례 검색 불완전 → **불편하지만 위험하지 않음** | **허용** |
| **이력형** (설정 변경 이력) | 최근 변경이 누락 → "왜 바꿨어?" 질의에 답 못 함 → **불편** | **허용** |

**2단계: 긴급도별 갱신 전략 결정**

| 갱신 긴급도 | → 갱신 방식 | 허용 지연 |
|:---------:|-----------|:--------:|
| **즉시** | 이벤트 트리거 (Webhook) | 수 분 이내 |
| **허용** | 이벤트 트리거 또는 스케줄 배치 | 수 시간~1일 |
| **느린** | 스케줄 배치 | 1일~1주 |

> 갱신 긴급도는 기술적 가능 여부가 아니라, **갱신 안 됐을 때 에이전트가 잘못된 판단을 내리는가** 로 결정한다.

**3단계: 파이프라인 구성**

긴급도가 결정되면 구체적인 파이프라인을 구성한다.

### 갱신 방식

| 갱신 방식 | 구현 | 적합 |
|---------|------|------|
| 이벤트 트리거 | Webhook / 파일 변경 감지 | 즉시 갱신이 필요한 컬렉션 |
| 스케줄 배치 | Cron / Airflow / n8n | 허용·느린 갱신이 가능한 컬렉션 |
| 수동 갱신 | 관리자 직접 트리거 | 연 1~2회 갱신되는 정적 문서 |

### 증분 업데이트 설계

| 작업 | 처리 방식 |
|------|---------|
| 새 문서 추가 | 파싱(Docling) → 청킹(HybridChunker) → 임베딩 → 해당 컬렉션에 삽입 |
| 기존 문서 수정 | 기존 청크 삭제(doc_id 기준) → 재파싱 → 재청킹 → 재삽입 |
| 문서 삭제 | doc_id 기준 해당 문서 전체 청크 삭제 |
| 메타데이터만 변경 | 벡터 유지, 메타데이터만 업데이트 (DB 지원 시) |

> **최신형 컬렉션 주의:** 문서 수정 시 구버전 청크 삭제가 반드시 선행되어야 한다. 삭제 없이 삽입만 하면 구버전과 신버전이 공존하여 검색 결과가 오염된다.

### 문서 ID ↔ 청크 ID 매핑

```
문서 고유 ID 체계:
```

### 실행 레이어 연동 (Phase 1 1-C, 1-D 연계)

| 연동 지점 | 대상 컬렉션 | 내용 |
|---------|-----------|------|
| 인시던트 대응 종결 (1-D) | incidents | JIRA 종결 → 자동 인덱싱 |
| 도구 신규 승인 완료 (1-C) | vendor_and_tools | 승인 목록 메타데이터 업데이트 |
| 정책 문서 갱신 | policy_and_guidelines | Webhook → 재인덱싱 |
| 외부 규제 변경 탐지 | policy_and_guidelines | 탐색(외부) 파이프라인 → 내부 문서 갱신 알림 → 수동 갱신 → Webhook |

### 갱신 모니터링 (검색 결과 신뢰도 중심)

| 모니터링 항목 | 의미 (검색 결과 관점) | 알림 조건 |
|------------|------------------|---------|
| 인덱싱 실패 건수 | 갱신된 문서가 반영 안 됨 → 구버전 히트 위험 | 1건 이상 즉시 알림 |
| 갱신 지연 시간 | 즉시 갱신 대상이 지연됨 → outdated 결과 반환 중 | 즉시 긴급도 컬렉션: 30분 초과 시 알림 |
| 벡터 수 vs 원본 문서 수 | 삭제 안 된 구버전 청크 잔존 → 구버전 히트 위험 | 주 1회 대조, 괴리 시 점검 |
| 인덱스 크기 급증 | 중복 삽입 또는 삭제 실패 → 노이즈 증가 | 주간 20% 이상 급증 시 알림 |
