## 개요

### GraphRAG란?

기존 RAG는 벡터 유사도 기반 검색이라 **의미 구분이 불가능**하다.

```
"사과" 검색 → 과일/기업/사죄 전부 유사도 비슷하게 나옴
```

GraphRAG는 벡터 검색에 **그래프 관계 탐색**을 추가하여 컨텍스트 기반 의미 파악이 가능하다.

```
"연체되면 한도에 영향있어?"
→ 벡터: 관련 문서 나열
→ GraphRAG: 연체 → 신용도 → 한도 관계 체인 추적
```

### 기술 스택

| 구성요소 | 기술 | 역할 |
|---------|------|------|
| 벡터 검색 | pgvector | 유사도 기반 문서 검색 |
| 그래프 탐색 | Apache AGE | 엔티티 관계 탐색 |
| DB | PostgreSQL 16 | 단일 DB에서 양쪽 처리 |
| 임베딩 | OpenAI text-embedding-3-small | 문서 임베딩 생성 |
| 엔티티 추출 | OpenAI gpt-4o-mini | 트리플 자동 추출 |

## §2. 엔티티+트리플 추출 파이프라인

> IRM v10 체크리스트 10번: 엔티티+트리플 추출 파이프라인

### 동작 방식

문서 하나당 병렬로 두 가지 처리:

```
문서 입력
├── 임베딩 생성 → pgvector 저장 (doc_id 생성)
└── 엔티티+트리플 추출 → AGE 노드+엣지 저장 (doc_id 참조)
```

두 작업은 독립적이며 doc_id로 연결된다.

### LLM 추출 프롬프트 설계

트리플 추출 품질은 프롬프트에 의존한다. 다음 규칙이 필수:

| 규칙 | 목적 | 위반 시 문제 |
|------|------|------------|
| from ≠ to (자기참조 금지) | 무의미한 루프 방지 | 그래프 탐색 시 무한 루프 |
| 관계명 동사형 강제 | 관계의 방향성 명확화 | "금리가" 같은 명사형은 관계 해석 불가 |
| 인과/조건 관계만 추출 | 의미 있는 관계만 보존 | 단순 나열 관계는 노이즈 |
| 엔티티 10개, 트리플 10개 이내 | 추출 비용 통제 | 과다 추출 시 그래프 오염 |
| 특수문자 금지 | Cypher 쿼리 파싱 보호 | SQL/Cypher 인젝션 위험 |

### 엔티티 중복 처리 (MERGE 패턴)

AGE 1.5.0은 `MERGE ON CREATE`를 지원하지 않는다. MATCH 후 존재 확인 → CREATE/UPDATE 분기로 처리한다.

```
신규 엔티티 → CREATE (name, doc_ids)
기존 엔티티 → SET doc_ids = doc_ids + ',{doc_id}'
```

### doc_id 연결 구조

벡터 청크(documents 테이블)와 그래프 노드(AGE entity)는 doc_id로 연결된다.

```
documents 테이블           AGE 그래프
┌─────────────────┐      ┌──────────────────┐
│ id (doc_id)     │◄────►│ entity.doc_ids   │
│ content         │      │ entity.name      │
│ meaning         │      └──────────────────┘
│ embedding       │              │
└─────────────────┘              │ 엣지 (relation)
                                 ▼
                         ┌──────────────────┐
                         │ entity.name      │
                         │ entity.doc_ids   │
                         └──────────────────┘
```

### 코드

```python
import openai
import psycopg2
import json
import os
from pgvector.psycopg2 import register_vector

os.environ["OPENAI_API_KEY"] = "API_KEY"
client = openai.OpenAI()

conn = psycopg2.connect(
    host="localhost",
    port=5556,
    dbname="graphrag",
    user="postgres",
    password="postgres"
)
register_vector(conn)
cur = conn.cursor()
cur.execute("LOAD 'age'")
cur.execute("SET search_path = ag_catalog, \"$user\", public")

# 기존 데이터 초기화
cur.execute("TRUNCATE TABLE documents RESTART IDENTITY")
cur.execute("""
    SELECT * FROM cypher('test_graph', $$
        MATCH (n) DETACH DELETE n
    $$) AS (result agtype)
""")
conn.commit()

def extract_graph(content):
    content = content.replace('"', '').replace("'", '').replace('\n', ' ')

    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {
                "role": "system",
                "content": """텍스트에서 엔티티와 관계를 추출해서 JSON으로만 반환해.
형식:
{
  "entities": ["엔티티1", "엔티티2"],
  "triples": [
    {"from": "엔티티1", "relation": "관계명", "to": "엔티티2"}
  ]
}
규칙:
- 엔티티는 명사, 10개 이내
- 트리플은 10개 이내
- from과 to는 반드시 달라야 함 (자기참조 절대 금지)
- 관계명은 반드시 동사형으로 (예: 영향받다, 발생하다, 결정하다, 포함하다)
- 관계명에 명사형 금지 (예: 금리가, 한도가, 상품 등)
- 인과관계/조건관계만 추출 (A가 B에 영향을 미치는 관계)
- 의미없는 관계 금지 (단순 언급, 나열 관계 금지)
- JSON만 반환, 다른 텍스트 없음
- 특수문자 사용 금지"""
            },
            {
                "role": "user",
                "content": content
            }
        ]
    )
    result = response.choices[0].message.content.strip()
    result = result.replace("```json", "").replace("```", "").strip()
    return json.loads(result)

# 문서 로드 (finance_docs.json)
with open("finance_docs.json", "r", encoding="utf-8") as f:
    data = json.load(f)

for doc in data["docs"]:
    try:
        # 1. 엔티티+트리플 추출
        graph = extract_graph(doc["content"])
        entities = graph["entities"]
        triples = graph["triples"]

        # 2. 임베딩 → pgvector 저장
        response = client.embeddings.create(
            input=doc["content"],
            model="text-embedding-3-small"
        )
        embedding = response.data[0].embedding
        cur.execute(
            "INSERT INTO documents (content, meaning, embedding) VALUES (%s, %s, %s) RETURNING id",
            (doc["content"], doc["domain"], embedding)
        )
        doc_id = cur.fetchone()[0]
        conn.commit()

        # 3. AGE 노드 생성 (MERGE 방식)
        for entity in entities:
            entity = entity.replace("'", "").replace('"', '').strip()
            if not entity:
                continue

            cur.execute(f"""
                SELECT * FROM cypher('test_graph', $$
                    MATCH (n:entity {{name: '{entity}'}})
                    RETURN n
                $$) AS (n agtype)
            """)
            existing = cur.fetchone()

            if existing:
                cur.execute(f"""
                    SELECT * FROM cypher('test_graph', $$
                        MATCH (n:entity {{name: '{entity}'}})
                        SET n.doc_ids = n.doc_ids + ',{doc_id}'
                        RETURN n
                    $$) AS (n agtype)
                """)
            else:
                cur.execute(f"""
                    SELECT * FROM cypher('test_graph', $$
                        CREATE (n:entity {{name: '{entity}', doc_ids: '{doc_id}'}})
                        RETURN n
                    $$) AS (n agtype)
                """)
            conn.commit()

        # 4. AGE 엣지 생성 (트리플)
        for triple in triples:
            f_node = triple["from"].replace("'", "").replace('"', '').strip()
            relation = triple["relation"].replace("'", "").replace('"', '').replace(' ', '_').strip()
            t_node = triple["to"].replace("'", "").replace('"', '').strip()

            if not f_node or not relation or not t_node:
                continue

            cur.execute(f"""
                SELECT * FROM cypher('test_graph', $$
                    MATCH (a:entity {{name: '{f_node}'}}), (b:entity {{name: '{t_node}'}})
                    CREATE (a)-[r:{relation} {{doc_id: {doc_id}}}]->(b)
                    RETURN r
                $$) AS (r agtype)
            """)
            conn.commit()

        print(f"완료: doc_id={doc_id} [{doc['domain']}] 엔티티:{len(entities)}개 트리플:{len(triples)}개")

    except Exception as e:
        print(f"에러: doc_id={doc.get('id')} {str(e)}")
        conn.rollback()
        continue

cur.close()
conn.close()
print("ingest_v3 완료!")
```

## §3. 그래프 검색 통합

> IRM v10 체크리스트 11번: 그래프 검색 통합

### 동작 방식

```
쿼리 입력
    ↓
1. 쿼리 임베딩 생성
    ↓
2. pgvector 벡터 검색 (top-k)
    ↓
3. 각 문서의 doc_id로 AGE 엔티티 조회
    ↓
4. 엔티티 기반 멀티홉 관계 탐색
    ↓
5. 관련문서 + 관계체인 컨텍스트 조립
    ↓
LLM에 전달할 준비 완료
```

### 멀티홉 탐색 깊이

| 홉 수 | Cypher 패턴 | 용도 |
|-------|-----------|------|
| 1홉 | `(a)-[r]-(b)` | 직접 관계 조회 |
| 2홉 (권장) | `(a)-[r1]-(b)-[r2]-(c)` | 인과 체인 탐색 |
| 3홉 | `(a)-[r1]-(b)-[r2]-(c)-[r3]-(d)` | 깊은 관계 탐색, 노이즈 주의 |

기본 권장은 **2홉**이다. 3홉 이상은 관계 체인이 길어지면서 노이즈가 증가하므로 도메인 특성에 따라 판단한다.

### doc_id 정확 매칭

doc_ids가 문자열 타입이므로 부분 매칭 오류를 방지해야 한다. `CONTAINS '9'`는 `'19'`, `'29'`도 히트한다.

```
정확 매칭 조건 (4가지 OR):
- doc_ids = '{id}'              → 단일 값
- doc_ids STARTS WITH '{id},'   → 첫 번째
- doc_ids ENDS WITH ',{id}'     → 마지막
- doc_ids CONTAINS ',{id},'     → 중간
```

### 컨텍스트 조립 형식

LLM에 전달하는 컨텍스트는 두 블록으로 구성:

```
[관련 문서]
- [{도메인}] 문서 내용

[관련 관계]
- 엔티티A -[관계1]-> 엔티티B -[관계2]-> 엔티티C
```

### REST API 통합 (단일 인터페이스)

IRM v10 원칙에 따라 탐색 레이어는 단일 엔드포인트로 노출한다.

```
POST /search {"query": "..."}
    → 내부: pgvector 벡터 검색 + AGE 그래프 탐색
    → 반환: {"context": "조립된 컨텍스트"}
```

호출하는 쪽은 내부 구현(RAG/GraphRAG)을 알 필요 없다. 초기(pgvector 단독)와 고도화(GraphRAG) 모두 동일 인터페이스를 유지한다.

### 코드

```python
import openai
import psycopg2
import os
from pgvector.psycopg2 import register_vector

os.environ["OPENAI_API_KEY"] = "API_KEY"
client = openai.OpenAI()

def get_db_connection():
    conn = psycopg2.connect(
        host="localhost",
        port=5556,
        dbname="graphrag",
        user="postgres",
        password="postgres"
    )
    register_vector(conn)
    cur = conn.cursor()
    cur.execute("LOAD 'age'")
    cur.execute("SET search_path = ag_catalog, \"$user\", public")
    return conn, cur

def graphrag_search(query, top_k=5):
    conn, cur = get_db_connection()

    # 1. 쿼리 임베딩
    response = client.embeddings.create(
        input=query,
        model="text-embedding-3-small"
    )
    query_embedding = response.data[0].embedding

    # 2. 벡터 검색
    cur.execute("""
        SELECT id, content, meaning, embedding <=> %s::vector AS distance
        FROM documents
        ORDER BY distance
        LIMIT %s
    """, (query_embedding, top_k))
    vector_results = cur.fetchall()

    # 3. 그래프 컨텍스트 조립
    docs_context = []
    graph_context = []

    for id, content, meaning, distance in vector_results:
        docs_context.append(f"- [{meaning}] {content}")

        # doc_id 정확 매칭 (부분 매칭 방지)
        cur.execute(f"""
            SELECT * FROM cypher('test_graph', $$
                MATCH (n:entity)
                WHERE n.doc_ids = '{id}'
                OR n.doc_ids STARTS WITH '{id},'
                OR n.doc_ids ENDS WITH ',{id}'
                OR n.doc_ids CONTAINS ',{id},'
                RETURN n.name
            $$) AS (name agtype)
        """)
        doc_entities = [e[0].strip('"') for e in cur.fetchall()]

        # 멀티홉 관계 탐색
        for entity in doc_entities[:3]:
            entity = entity.replace("'", "")
            cur.execute(f"""
                SELECT * FROM cypher('test_graph', $$
                    MATCH (a:entity {{name: '{entity}'}})-[r1]-(b:entity)-[r2]-(c:entity)
                    WHERE a <> c
                    RETURN a.name, type(r1), b.name, type(r2), c.name
                    LIMIT 3
                $$) AS (a agtype, r1 agtype, b agtype, r2 agtype, c agtype)
            """)
            hops = cur.fetchall()
            for h in hops:
                a = h[0].strip('"')
                r1 = h[1].strip('"')
                b = h[2].strip('"')
                r2 = h[3].strip('"')
                c = h[4].strip('"')
                graph_context.append(f"- {a} -[{r1}]-> {b} -[{r2}]-> {c}")

    # 중복 제거
    graph_context = list(set(graph_context))

    # 4. LLM 컨텍스트 조립
    assembled = f"""[관련 문서]
{chr(10).join(docs_context)}

[관련 관계]
{chr(10).join(graph_context[:10])}
"""

    cur.close()
    conn.close()
    return assembled


# 실행
query = "연체되면 대출 한도에 영향있어?"
context = graphrag_search(query)
print(f"=== 쿼리: '{query}' ===")
print(context)
```

## §4. 갱신 파이프라인

> IRM v10 체크리스트 12번: GraphRAG 갱신 파이프라인

### 갱신 원칙

GraphRAG는 벡터(pgvector)와 그래프(AGE) 두 저장소를 동시에 관리해야 한다. **한쪽만 갱신하면 정합성이 깨진다.**

```
문서 변경 발생
    ↓
벡터 저장소 갱신 (pgvector)
    + 동시에
그래프 저장소 갱신 (AGE)
    ↓
정합성 확인
```

### 갱신 유형별 처리

| 갱신 유형 | pgvector 처리 | AGE 처리 |
|----------|-------------|---------|
| 신규 문서 | INSERT (임베딩 + doc_id 발급) | 엔티티 MERGE + 엣지 CREATE |
| 문서 수정 | UPDATE (임베딩 재생성) | 기존 엣지 DELETE + 재추출 + 재생성 |
| 문서 삭제 | DELETE | 해당 doc_id 엣지 DELETE + 고아 노드 정리 |

### 신규 문서 갱신

§2 인제스트 파이프라인과 동일한 흐름이다. 기존 데이터 초기화 없이 추가만 수행한다.

### 문서 수정 갱신

문서 내용이 변경되면 엔티티/트리플도 달라질 수 있다. 임베딩 재생성과 그래프 재추출을 동시에 수행한다.

```
1. documents 테이블에서 해당 doc_id의 임베딩 UPDATE
2. AGE에서 해당 doc_id의 엣지 DELETE
3. 엔티티의 doc_ids에서 해당 doc_id 제거
4. 변경된 내용으로 트리플 재추출
5. 재추출된 엔티티/엣지 재생성
```

### 문서 삭제 갱신

```
1. documents 테이블에서 해당 doc_id DELETE
2. AGE에서 해당 doc_id의 엣지 DELETE
3. 엔티티의 doc_ids에서 해당 doc_id 제거
4. doc_ids가 빈 문자열이 된 고아 노드 DELETE
```

### 고아 노드 정리

문서 삭제/수정 시 어떤 문서와도 연결되지 않은 엔티티가 남을 수 있다. 주기적으로 정리한다.

```sql
-- doc_ids가 비어있는 고아 노드 조회
SELECT * FROM cypher('test_graph', $$
    MATCH (n:entity)
    WHERE n.doc_ids = '' OR n.doc_ids IS NULL
    RETURN n.name
$$) AS (name agtype);

-- 고아 노드 삭제
SELECT * FROM cypher('test_graph', $$
    MATCH (n:entity)
    WHERE n.doc_ids = '' OR n.doc_ids IS NULL
    DETACH DELETE n
$$) AS (result agtype);
```

### 갱신 트리거 방식

IRM v10 RAG 설계 방법론 §8의 갱신 긴급도 분류를 따른다.

| 긴급도 | 트리거 방식 | 예시 |
|-------|-----------|------|
| 즉시 | 이벤트 트리거 (문서 저장 시 즉시 실행) | 장애 포스트모템 |
| 허용 | 스케줄 배치 (일배치/시간배치) | 정책 문서, 매뉴얼 |
| 느린 | 수동 트리거 | 아키텍처 문서 |

### 추출 비용 관리

LLM 기반 트리플 추출은 문서당 API 호출이 발생한다. 대량 갱신 시 비용을 통제해야 한다.

| 전략 | 방법 |
|------|------|
| 로컬 LLM 교체 | OpenAI → Ollama (폐쇄망 대비 겸용) |
| 변경 감지 | 문서 해시 비교로 실제 변경분만 재추출 |
| 배치 처리 | 즉시 갱신이 아닌 경우 배치로 묶어서 처리 |

## 테스트 결과

### 순수 벡터 RAG vs GraphRAG 비교

**쿼리: "사과"**

```
순수 벡터 RAG:
[사죄] 거리:0.7274
[과일] 거리:0.7382
[사죄] 거리:0.7564
→ 동음이의어 구분 불가, 왜 이게 나왔는지 설명 불가
```

```
GraphRAG:
[사죄] 엔티티: 실수, 사과, 상대방
[과일] 엔티티: 사과, 농장, 수확량, 과수원
→ 엔티티로 의미 구분 가능, 맥락 추적 가능
```

**쿼리: "연체되면 대출 한도에 영향있어?"**

```
GraphRAG 관계 체인:
Y은행 대출 -[영향받다]-> 신용도 -[영향미치다]-> 연체
차세대 대출 상품 -[포함하다]-> 한도 -[영향받다]-> 신용도
신용도 -[영향미치다]-> 연체 -[발생하다]-> 손해
→ 연체-신용도-한도 인과관계 체인 자동 탐색
```

### 증명된 것

| 항목 | 결과 |
|------|------|
| 동음이의어 구분 | 벡터 불가 / GraphRAG 가능 |
| 맥락 설명 가능 여부 | 벡터 불가 / GraphRAG 가능 |
| 멀티홉 2단계 탐색 | 동작 확인 |
| 멀티홉 3단계 탐색 | 동작 확인 |
| LLM 컨텍스트 조립 | 완성 |

## IRM 적용

### IRM 탐색 영역 표준

```
탐색(내부) 구현 표준:
    기존: RAG (pgvector 단독)
    고도화: GraphRAG (pgvector + Apache AGE)
```

### 적용 기준

| 상황 | 선택 |
|------|------|
| 단순 키워드 검색 | pgvector 단독 |
| 동음이의어 많은 도메인 | GraphRAG |
| 관계/인과 탐색 필요 | GraphRAG |
| 공공기관/금융 도메인 | GraphRAG 권장 |

### GraphRAG가 IRM에서 하는 역할

```
탐색 레이어:
    pgvector → 유사 문서 top-k 검색
    Apache AGE → 관계 체인으로 컨텍스트 보강
    조립된 컨텍스트 → 실행 레이어로 전달

실행 레이어 (별도):
    GraphRAG 결과를 컨텍스트로 받아서 실행 판단
```

GraphRAG는 **탐색 표준**이다. 실행 레이어와 역할이 명확히 분리된다.

## 주요 이슈 및 해결

| 이슈 | 원인 | 해결 |
|------|------|------|
| AGE LOAD 필요 | 매 커넥션마다 필요 | `cur.execute("LOAD 'age'")` 추가 |
| Cypher 파라미터 바인딩 불가 | AGE 제약 | f-string으로 직접 삽입 |
| doc_ids 부분 매칭 오류 | CONTAINS '9' → '19','29' 히트 | STARTS WITH / ENDS WITH / CONTAINS ',n,' 조합 |
| AGE MERGE ON CREATE 미지원 | AGE 1.5.0 제약 | MATCH 후 존재 확인 → CREATE/UPDATE 분기 |
| 자기참조 트리플 | LLM 추출 품질 | 프롬프트에 자기참조 금지 규칙 추가 |
| 트리플 관계명 명사형 | LLM 추출 품질 | 프롬프트에 동사형 강제 규칙 추가 |
| 동의어 엔티티 분산 | LLM 자유 추출 특성 | 같은 개념이 "금리"/"이자율"/"금리인상" 등으로 분산 추출됨. 현재 미해결 — 향후 정규화 필요 |

## 향후 개선사항

- **엔티티 정규화**: 동의어/유사어 병합 (금리↔이자율, 연체↔연체발생 등). 자유 추출 방식의 구조적 한계로, 축적된 엔티티를 사후 클러스터링하여 정규화하는 단계 필요. bottom-up 온톨로지 구축의 시작점
- **doc_ids 타입 개선**: 문자열 → 배열 타입으로 변경 (부분 매칭 이슈 근본 해결)
- **로컬 임베딩 모델 교체**: OpenAI → Ollama (폐쇄망 환경 대비)
- **트리플 추출 비용 최적화**: 로컬 LLM + 병렬 처리
- **REST API 래핑**: `POST /search` 엔드포인트로 표준화
