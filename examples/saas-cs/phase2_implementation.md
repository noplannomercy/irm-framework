# Phase 2 — 구현 체크리스트

> 도메인: SaaS 고객 성공팀
> 단계: Phase 2
> 상태: 완료
> 입력: phase1_1a ~ 1e 전체

---

## 개요

Phase 1의 배치 대상과 요구사항을 기반으로, 실제 구현 시 확인해야 할 의사결정 항목을 체크리스트로 정리한다.
"어떻게 결정하는가"는 IRM Framework v7 별첨 방법론을 참조한다.

---

## 구현 우선순위 요약

Phase 1 전체에서 도출된 MCP 승격 우선순위와 체이닝 의존관계를 고려한 구현 순서:

| 순서 | 대상 | 근거 |
|-----|-----|-----|
| 1 | 고객 컨텍스트 패키지 MCP (CRM+Chargebee+Analytics) | 모든 실행 체이닝의 선행 조건, ★★★ |
| 2 | RAG 구축 (CS 대응 사례 + 제품 가이드 + 릴리즈노트 + 절차 문서) | 탐색 단독 질의 + 실행(유연) 체이닝의 선행 조건 |
| 3 | 실행(확정) MCP 파이프라인 (온보딩, 갱신·해지, 인보이스 처리) | 고객 컨텍스트 패키지 MCP 완료 후 연결 |
| 4 | 실행(확정) MCP 파이프라인 (에스컬레이션, 계정정지·해제) | Zendesk·Slack·AdminAPI MCP 완료 후 연결 |
| 5 | 실행(유연) Workflow (이탈 위기 대응, 갱신 미팅 준비) | RAG + 고객 컨텍스트 패키지 MCP 완료 후 구성 |
| 6 | 나머지 시스템 MCP (Zendesk, Gmail, Slack, AdminAPI, NPS) | 실행(확정) 파이프라인 연결 시점에 순차 구현 |

### 탐색(내부) — RAG 구축

- [ ] **1. 컬렉션 설계**
  - 갱신 주기와 검색 패턴 기준 컬렉션 분리 제안:
    - `product` — 제품 가이드(#1), 릴리즈 노트(#3), 가격정책(#8) / 기능 출시 시 갱신
    - `cs-cases` — CS 대응 사례(#4), 버그·이슈 이력(#2) / 케이스 입력 시 갱신
    - `policy` — SLA 기준(#6), 에스컬레이션 정책(#7), 온보딩 가이드(#5) / 정책 변경 시 갱신
    - `runbook` — 절차 문서 #17~22 / 수시 갱신
  - 벡터 저장소 제품 선정
  - → 별첨: RAG 설계 방법론 §1

- [ ] **2. 컬렉션별 메타데이터 스키마 확정**
  - 공통: `doc_id`, `source_url`, `updated_at`, `collection`
  - `product`: `feature_name`, `category`, `plan_availability`, `version`, `change_type`
  - `cs-cases`: `case_type`, `customer_plan`, `feature`, `resolution_path`, `issue_type`, `resolution`
  - `policy`: `doc_type`, `scope`, `metric_type`, `plan_tier`, `threshold`
  - `runbook`: `procedure_name`, `trigger_condition`, `step_count`
  - → 별첨: RAG 설계 방법론 §2

- [ ] **3. 문서 파싱 및 청킹**
  - Notion 커넥터 연동 방식 결정 (Notion API 직접 vs 서드파티 커넥터)

  - 절차 문서(runbook) 청킹 단위: 스텝 단위 vs 문서 전체
  - → 별첨: RAG 설계 방법론 §3

- [ ] **4. 임베딩 모델 선정**
  - 한국어 지원 여부 확인 (CS 사례가 한국어인 경우)
  - 로컬 vs API 기반 결정
  - → 별첨: RAG 설계 방법론 §4

- [ ] **5. 검색 전략 결정**
  - 하이브리드 검색 적용 여부 (`cs-cases`는 키워드+벡터 조합 권장)
  - 컬렉션별 메타데이터 필터링 적용:
    - `product`: `plan_availability` 필터 (고객 플랜 기준 결과 제한)
    - `cs-cases`: `case_type`, `feature` 필터
    - `policy`: `plan_tier` 필터 (SLA는 플랜별 기준 다름)
  - → 별첨: RAG 설계 방법론 §5

- [ ] **6. 리랭킹 적용 여부**
  - `cs-cases` 컬렉션은 유사 사례 정밀도가 핵심 — 리랭킹 우선 적용 검토
  - → 별첨: RAG 설계 방법론 §6

- [ ] **7. 인덱싱 범위 결정**
  - `cs-cases`: 이력형 (과거 사례 전체 포함)
  - `product`: 최신형 (현재 버전 가이드만, 구버전 삭제)
  - `policy`: 최신형 (현행 정책만)
  - `runbook`: 최신형 (현행 절차만)
  - → 별첨: RAG 설계 방법론 §7

- [ ] **8. 갱신 파이프라인 설계**
  - `cs-cases`: CS팀이 케이스 입력 시 즉시 갱신 (이벤트 트리거)
  - `product`: 릴리즈 배포 시 트리거 연동
  - `policy`: 수동 갱신 허용 (낮은 갱신 빈도)
  - `runbook`: 문서 변경 시 스케줄 배치 또는 수동
  - → 별첨: RAG 설계 방법론 §8

## 2. 접근 구현 체크리스트

**입력: phase1_1b_access.md 1-B-2**

- [ ] **1. MCP 대상 확정**
  - ★★★ 즉시 MCP로 시작: CRM(#9), Chargebee(#10), Analytics(#11)
  - ★★ 운영 중 승격: Zendesk(#12), Gmail(#13), Admin API(#16)
  - ★ 승격 대기: Slack(#14), NPS(#15)
  - → 별첨: MCP 승격 설계 가이드 §1

- [ ] **2. MCP 서버 경계 결정**
  - **고객 컨텍스트 패키지 — Aggregator 패턴 적용:**
    - CRM + Chargebee + Analytics를 Aggregator MCP로 감쌈
    - 에이전트는 Aggregator 하나만 알면 됨. 안쪽 3개 시스템은 독립 MCP 유지
    - 근거: 항상 세트로 호출됨, 고객 ID 기준 조회, 모든 실행 체이닝의 선행 구간
  - Zendesk, Gmail, Slack, AdminAPI: 각각 개별 MCP 서버 (시스템 기준 분리 원칙)
  - Admin API: 계정 정지·해제 파이프라인(1-C)과 경계 결정
  - **컨텍스트 비용 확인**: 이 도메인 MCP tool 총 수가 50개 이하인지 확인
  - → 별첨: MCP 승격 설계 가이드 §2 (특히 §2-0, §2-1, §2-3)

- [ ] **3. 도구 목록 및 그루핑 결정**
  - **MCP는 얇게, CLI는 두텁게** 원칙 적용:
    - 고객 컨텍스트 패키지 도구는 3~5개로 최소화
    - `get_customer_context(customer_id)` — CRM+Chargebee+Analytics 통합 조회 (CLI가 내부에서 조합)
    - `get_subscription_status(customer_id)` — 구독 상태 단독 조회
    - `get_usage_trend(customer_id, period)` — 사용량 추이 조회
  - REST API 1:1 래핑 금지 — 업무 의미 단위로 설계. 세부 기능은 hubspot-cli, chargebee-cli 등으로
  - 도구 수 기준: 서버당 5~8개 이상적, 12개 초과 시 재검토
  - → 별첨: MCP 승격 설계 가이드 §2-2, §3

- [ ] **4. Transport 결정**
  - CS팀 전원 공유 → HTTP (Streamable HTTP) 방향
  - 초기 개발·테스트는 stdio로 시작
  - → 별첨: MCP 승격 설계 가이드 §4

- [ ] **5. mcp-builder로 구현**
  - 위 1~4 결정사항 전달
  - 결과물 검증: 도구가 "조회 단위"가 아닌 "업무 의미 단위"로 설계됐는지 확인
  - → 별첨: MCP 승격 설계 가이드 §5

- [ ] **6. MCP 통합 검증**
  - 단독 조회 질의(#1~5)로 각 MCP 도구 동작 확인
  - 체이닝 진입 검증: 실행(확정) 파이프라인 앞단에서 고객 컨텍스트 정상 반환 확인

## 도메인별 주의 사항
