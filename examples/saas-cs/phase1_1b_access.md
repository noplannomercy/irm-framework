## 체이닝 참여 분석

접근 단독으로 끝나는 질의는 5개(#1~5)이고, 나머지는 실행(확정/유연)과 체이닝된다. 체이닝 참여 패턴에서 핵심 시스템이 드러난다.

| 체이닝 유형 | 질의 예 | 접근 구간 시스템 |
|-----------|-------|--------------|
| 접근+실행(확정) | 온보딩(#16), 갱신/해지(#17,#18), 계정정지(#19), 인보이스오류(#20), 에스컬레이션(#21) | #9,#10,#11,#12,#13,#14,#16 |
| 접근+탐색(내부)+실행(유연) | 이탈위기 대응(#22), 갱신 미팅 준비(#23) | #9,#10,#11 |
| 접근+실행(유연) | 사용량 낮은 고객 대응(#24), CS 현황 리포트(#25) | #9,#10,#11,#12,#15 |

> **핵심 패턴:** CRM(#9) + 구독관리(#10) + 사용량모니터링(#11)은 사실상 모든 실행 체이닝에 공통으로 포함된다. 이 세 시스템을 하나의 "고객 컨텍스트 패키지"로 묶어 **Aggregator 패턴** MCP로 구현하는 것이 효율적이다.
>
> **Aggregator 패턴 적용:**
> ```
> Agent
>   └── Customer Context Aggregator MCP (얇은 창구, 3~5개 tool)
>         ├── → CRM MCP (HubSpot) — 독립 유지
>         ├── → Chargebee MCP — 독립 유지
>         └── → Analytics MCP — 독립 유지
> ```
> 에이전트는 Aggregator 하나만 알면 되고, `get_customer_context(customer_id)` 한 번 호출로 세 곳 데이터를 반환한다. 각 시스템 MCP는 내부에서 독립적으로 유지하여 장애 격리·독립 배포를 보장한다.

## 성숙도 메모

| 항목 | 초기 구현 | 성숙 시 |
|-----|---------|--------|
| CRM, 구독관리, 사용량모니터링 | 직접 API 호출 | **Aggregator 패턴** MCP (고객 컨텍스트 패키지) — 최우선. Tool 3~5개. 세부 기능은 CLI |
| 티켓, 이메일, Slack | 직접 API 호출 | MCP 접근형 (실행 체이닝 시 자연스럽게 승격). MCP는 얇게, CLI는 두텁게 |
| 계정 관리 콘솔 | 직접 API 호출 | MCP 접근형 + 계정 정지/해제는 실행(확정) 1-C와 묶어서 파이프라인형 검토 |
| NPS/CSAT | 직접 API 호출 | 승격 우선순위 낮음 — 조회 빈도 확인 후 결정 |

> **MCP 설계 핵심 원칙 (이 도메인 적용):**
> - 고객 컨텍스트 패키지는 시스템 3개이지만 외부 노출 tool은 3~5개가 적정
> - CRM/Chargebee/Analytics 각각 수십 개의 API 엔드포인트가 있어도 MCP에는 업무 의미 단위만 올린다
> - Tool 컨텍스트 비용(1개당 550~1,400 토큰) 고려 — 에이전트 전체 50개 상한 내에서 설계
> - → 별첨: MCP 승격 설계 가이드 §2-0, §2-1, §2-2, §2-3
