# Phase 1 1-E — 체이닝 관계 맵

> 도메인: SaaS 고객 성공팀
> 단계: Phase 1 1-E
> 상태: 완료
> 입력: phase1_1a ~ 1d 전체

---

## 개요

1-A~1-D에서 정의한 배치 대상들이 실제 질의 처리 시 어떻게 연결되는지를 체이닝 관계로 정리한다.
체이닝은 하나의 질의가 여러 행위 유형을 순서대로 거치는 흐름이다.

---

## 공통 접근 구간 — 고객 컨텍스트 패키지

실행을 포함하는 질의(#15~25) 중 **#15, #22~25를 제외한 모든 케이스**가 CRM(#9) 또는 Chargebee(#10) 또는 Analytics(#11) 중 하나 이상을 접근 구간에서 반드시 거친다.

```
[고객 컨텍스트 패키지] — MCP 접근형 최우선 승격 대상
  CRM (#9)         — 고객 프로필, 계약 현황
  Chargebee (#10)  — 구독 상태, 결제, 인보이스
  Analytics (#11)  — 사용량, 로그인 이력
```

이 세 시스템이 항상 세트로 조회되는 이유:
- 실행(확정) 절차 진입 전에 "현재 상태 확인"이 필수
- 실행(유연)에서 LLM이 판단하려면 고객 전체 맥락이 필요

## 행위 유형별 컴포넌트 연결 구조

```
탐색(내부)
  └── RAG
        ├── 제품/버그/릴리즈/사례/온보딩/SLA/에스컬레이션정책/가격정책 (#1~8)
        └── 절차 문서 지식 (#17~22)

탐색(외부)
  └── 웹 검색
        ├── 고객사 IR (#23)
        ├── CS 벤치마크 (#24)
        └── 경쟁사 비교 (#25)

접근
  └── 고객 컨텍스트 패키지 (MCP 접근형)
        ├── CRM (#9)
        ├── Chargebee (#10)
        └── Analytics (#11)
  └── 실행 연계 시스템 (MCP 접근형)
        ├── Zendesk (#12)
        ├── Gmail (#13)
        ├── Slack (#14)
        ├── NPS (#15)
        └── Admin API (#16)

실행(확정)
  └── MCP 파이프라인형
        ├── 온보딩 절차 (#17) → CRM + Gmail
        ├── 갱신·해지 절차 (#18) → Chargebee + Gmail
        ├── 계정정지·해제 절차 (#19) → Admin API + Gmail
        ├── 에스컬레이션 절차 (#21) → Zendesk + Slack
        └── 인보이스 오류 절차 (#22) → Chargebee + Gmail

실행(유연)
  └── Workflow → MCP 에이전트형
        ├── 이탈 위기 대응 (#20a) → 고객 컨텍스트 패키지 + RAG
        ├── 갱신 미팅 준비 (#20b) → 고객 컨텍스트 패키지 + RAG + 웹검색
        ├── 사용량 낮은 고객 대응 → 고객 컨텍스트 패키지
        └── CS 현황 리포트 → 고객 컨텍스트 패키지 + Zendesk + NPS
```

## 체이닝 흐름에서 드러나는 설계 시사점

**1. Gmail이 모든 실행(확정)의 끝에 붙는다**
온보딩·갱신·해지·계정정지·인보이스 — 5개 절차 모두 Gmail 발송으로 마무리된다.
→ Gmail MCP는 실행(확정) 파이프라인의 공통 마지막 단계로 설계하는 것이 효율적.

**2. 이탈 위기 대응은 가장 복잡한 체이닝**
접근(CRM+Analytics) → 탐색(RAG: CS사례+플레이북) → 실행(유연) 세 구간을 모두 거친다.
→ MCP 에이전트형으로 전환할 때 탐색 구간(RAG 호출)도 MCP 내부에서 처리할 수 있어야 한다.

**3. 실행(확정)은 접근 구간과 강하게 결합**
실행(확정) 5개 모두 접근 구간을 선행한다. 접근 MCP가 준비되지 않으면 실행(확정) MCP 파이프라인을 구현할 수 없다.
→ 구현 순서: 접근 MCP 먼저 → 실행(확정) MCP 파이프라인 연결.

**4. 탐색(외부)은 실행(유연)에서만 등장**
외부 정보(고객사 IR, 경쟁사 비교)는 갱신 미팅 준비(#23)와 이탈 위기 대응 간접 참조에서만 쓰인다.
→ 탐색(외부) 인프라는 실행(유연) 파이프라인 구축 시점에 같이 고려하면 충분.
