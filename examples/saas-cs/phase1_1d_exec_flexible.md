## 1-D-2: 실행(유연) 요구사항 도출

실행(유연)은 입출력이 고정되지 않는다. 대신 **LLM이 판단해야 할 분기 지점**과 **필요한 컨텍스트**를 정의한다.

| 절차 | 필요한 컨텍스트 | LLM 판단 분기 | 가능한 출력 범위 | MCP 에이전트형 승격 우선순위 |
|-----|------------|------------|--------------|:----------------------:|
| 이탈 위기 대응 | 구독 상태, 사용량 추이, NPS, 미결 티켓, 유사 CS 사례 | 이탈 원인 유형 분류 → 원인별 대응 전략 선택 (가격 문제/기능 문제/지원 문제 등) | 대응 플랜 제안, 미팅 요청 메일 초안, 할인 제안 여부 | ★★★ |
| 갱신 미팅 준비 | 구독 만료일, 사용량, NPS, 플랜 기능, 고객사 IR | 리스크 레벨 판단 → 미팅 전략 선택 (유지/업셀/위험 고객 집중 관리) | 미팅 아젠다, 리스크 요약, 업셀 포인트, 레퍼런스 자료 | ★★ |
| 사용량 낮은 고객 대응 | 사용량 시계열, 고객 플랜, 온보딩 완료 여부 | 저사용 원인 추정 (온보딩 미완/기능 미인지/이탈 전조) → 고객 세그먼트별 대응 방식 결정 | 세그먼트별 대응 리스트, 개인화 메일 초안 | ★★ |
| CS 현황 리포트 | 티켓 수/해결률, 사용량 지표, NPS/CSAT, 갱신 현황 | 핵심 이슈 우선순위 판단 → 리포트 구성 방식 결정 | 리포트 초안 (요약+지표+인사이트), 액션 아이템 | ★ |

## v7 논리: Workflow → MCP 에이전트형 흡수

| 항목 | 현재 (Workflow) | 성숙 시 (MCP 에이전트형) |
|-----|--------------|---------------------|
| 이탈 위기 대응 | n8n 등으로 데이터 수집 → LLM 판단 → 다음 단계 분기 | MCP 에이전트형: LLM이 판단, 실행만 MCP로 — 각 단계가 CLI 명령어로 기록됨 |
| 갱신 미팅 준비 | 여러 시스템 순차 조회 → LLM이 구성 | MCP 에이전트형: 필요한 데이터를 LLM이 선택적으로 조회하며 구성 |
| 사용량 낮은 고객 대응 | 사용량 데이터 조회 → 세그먼트 분류 → 대응 | MCP 에이전트형: 세그먼트 판단 + 고객별 대응 실행 혼재 |

> **v7 핵심:** Workflow에서는 "LLM 판단 → 즉시 외부 시스템 호출"이 추적 없이 일어난다. MCP 에이전트형으로 전환하면 판단은 LLM이 하되, 실행은 MCP(CLI)로만 하므로 각 단계가 기록·재현 가능해진다.

## 실행(유연) → 실행(확정) 전환 가능성

실행(유연)은 반복 운영하면서 패턴이 고정되는 구간이 생기면 실행(확정)으로 전환할 수 있다.

| 절차 | 전환 가능 구간 | 전환 조건 |
|-----|------------|---------|
| 이탈 위기 대응 | "가격 문제 → 할인 제안 + 미팅 요청" 패턴 고정 시 | 원인 유형별 대응이 표준화되면 1-C로 이동 |
| 갱신 미팅 준비 | 미팅 아젠다 구성 포맷이 고정될 때 | 출력 형식만 표준화 → Skills(프롬프트) 승격 가능한 유일한 후보 |
| 사용량 낮은 고객 대응 | 세그먼트 분류 기준이 안정화될 때 | 분류 로직 고정 → 세그먼트별 대응은 1-C로 이동 |

> **갱신 미팅 준비**는 이 도메인에서 "순수 프롬프트(출력 형식 표준화)"로 Skills 승격이 가능한 유일한 후보다. 데이터 수집은 MCP(접근), 분석·구성은 LLM, 포맷 표준화는 Skills로 역할 분리가 명확해지는 시점에 분리한다.
