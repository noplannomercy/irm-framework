## 체이닝 관계 맵

| # | 질의 | 체이닝 순서 | GraphRAG |
|---|------|-----------|:--------:|
| 1 | 수입검사 진행해줘 | 접근(ERP) → 탐색(기준서) → 접근(LIMS) → 실행(확정) → 접근(품질DB 기록) | 불필요 |
| 2 | 공정검사 진행해줘 | 접근(MES) → 탐색(기준서) → 실행(확정) → 접근(품질DB 기록) | 불필요 |
| 3 | 출하검사 진행해줘 | 접근(품질DB) → 접근(LIMS) → 탐색(기준서) → 실행(확정) → 접근(품질DB 기록) | 불필요 |
| 4 | 검사 성적서 발행해줘 | 접근(품질DB+LIMS) → 실행(확정) | 불필요 |

---

## 주목할 패턴

**1. 전부 GraphRAG 불필요**
탐색이 "기준서 룩업"으로 한정. 메타데이터 필터링(product, inspection_type)이면 충분.

**2. 접근이 파이프라인의 대부분**
각 검사 절차가 2~3개 시스템을 순차 호출. CLI 파이프라인형이 자연스러움.

**3. 유즈케이스 1 체이닝과 완전히 다름**

| | 유즈케이스 1 (불량 분석) | 유즈케이스 2 (검사 절차) |
|---|---|---|
| 패턴 | 접근 → **탐색(GraphRAG)** → 실행(유연) | 접근 → 탐색(룩업) → **실행(확정)** → 접근(기록) |
| LLM 역할 | 판단 (원인 추론) | 파이프라인 제어 (순서 실행) |
| GraphRAG | 필수 | 불필요 |
