## GraphRAG 승격 판단 (v10)

> v10 기준: 4개 조건 중 **2개 이상 충족 시** GraphRAG 승격을 검토한다.

| # | 승격 조건 | 충족 여부 | 근거 |
|---|---------|:-------:|------|
| 1 | 동음이의어/중의어가 많다 | **O** | "불량"(수입불량/공정불량/출하불량/고객불량), "기준"(검사기준/관리기준/판정기준), "공정"(사출공정/조립공정/도장공정) — 같은 단어가 맥락마다 다른 의미 |
| 2 | 관계/인과 기반 질의 패턴 발견 | **O** | #6 "비슷한 불량 전에 있었어?", #7 "원인이 뭐였어?", #22 "원인 분석해줘" — 불량→원인→공정→설비→자재 인과 체인. 전체 탐색 질의 중 인과 패턴 약 30% |
| 3 | RAG 히트율 목표치 미달 | — | 운영 전이므로 판단 불가 |
| 4 | 동일 엔티티가 여러 도메인에 다른 맥락으로 등장 | **O** | "불량"이 수입검사/공정검사/출하검사/고객클레임에서 각각 다른 원인·처리 경로를 가짐. "기준"이 검사기준서/SPC관리기준/출하판정기준으로 분산 |

**충족: 3/4** → **GraphRAG 승격 검토 대상**

**판단:** 제조/품질관리는 인과관계 추적이 업무 핵심이다. 불량→원인→공정→설비→자재 체인을 그래프로 탐색하면 "비슷한 불량"을 벡터 유사도가 아닌 관계 구조로 찾을 수 있다. 운영 초기부터 GraphRAG 적용을 권장하되, "RAG 먼저 → 필요 시 승격" 원칙을 따르는 것이 표준이다.

> **참조:** IRM Framework v10 — 성숙도 승격 기준 / GraphRAG 설계 방법론 v10

## 컬렉션 설계 제안

```
quality-standards   — 검사 기준서, 제품 규격서, SPC 관리 기준서, 공정 매뉴얼
                     갱신: 기준/규격 변경 시 / 검색 패턴: 제품·공정·검사유형 기준
                     메타데이터: product, process, inspection_type, spec_version

defect-history     — 고객 클레임 이력, 불량 대응 이력 (품질DB 연동)
                     갱신: 클레임/불량 접수 시 / 검색 패턴: 불량유형·제품·원인·기간
                     메타데이터: defect_type, product, root_cause, period, severity

vendor-quality     — 공급업체 평가 이력
                     갱신: 평가 시 / 검색 패턴: 업체명·평가항목·기간
                     메타데이터: vendor_name, eval_type, period

runbook            — 절차서 #16~20, 불량 대응 매뉴얼, CAPA 가이드 (지식 용도)
                     갱신: 절차 변경 시 / 검색 패턴: 절차명·트리거·불량유형
                     메타데이터: procedure_name, trigger_condition, defect_type
```

> **4개 컬렉션 기준:** 갱신 주기(기준서/클레임/평가/절차)와 검색 패턴(규격조회/불량탐색/업체비교/절차참조)이 컬렉션마다 다르다.
>
> **finance-procurement과 비교:** 둘 다 GraphRAG 승격 대상이지만, 제조는 "불량→원인→공정" 인과 체인이 핵심이고, 재무는 "승인" 동음이의어가 핵심. 그래프 구조가 다르다.
