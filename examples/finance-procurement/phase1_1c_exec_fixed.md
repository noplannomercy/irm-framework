## 설계 시사점

**1. 구매 전체 프로세스 (#21)가 메타 파이프라인**
#15 → #16 → #17 → #18 네 절차가 #21의 서브 파이프라인으로 동작.
→ HR 온보딩 Day1과 동일한 계층적 파이프라인 구조.
→ 구매 MCP가 각 서브 파이프라인 MCP를 순서대로 호출하는 계층 구조 설계 필요.

**2. 전자결재(#10)가 모든 실행(확정)의 필수 중간 단계**
구매 요청(#15), 발주서 발행(#16, 선행 승인 확인), 지급 처리(#18), 결재 상신(#20) — 전자결재 없이는 어떤 절차도 진행 불가.
→ 전자결재는 독립 MCP tool이 아니라 파이프라인 내 **승인 게이트 단계**로 내장 설계해야 함.
→ 승인 대기 상태에서 파이프라인이 멈춰야 하므로, **비동기 처리 + 상태 추적** 필요.

**3. ERP가 모든 실행(확정)의 첫 번째 단계**
7개 절차 대부분이 ERP 조회·생성으로 시작.
→ ERP MCP가 준비되지 않으면 어떤 실행(확정) 파이프라인도 구현 불가.
→ 구현 순서: 구매 컨텍스트 패키지 Aggregator MCP 먼저, 파이프라인 연결 그 다음.

**4. DMS가 실행(확정) 4개의 공통 마지막 단계**
발주서 발행(#16), 검수 처리(#17), 지급 처리(#18), 법인카드 정산(#19) 모두 DMS 증빙 등록으로 마무리.
→ HR 온보딩의 Gmail 역할. DMS MCP를 공통 마지막 단계로 재사용 설계.

**5. 지급 처리(#18) 승인 게이트 여부 결정 필요**
실제 금액 이체 포함. 오발동 시 금전 피해 발생.
→ 자동 실행 vs human-in-the-loop 확인 필요 → **현업 결정 필요** ⚠️

**6. 구매 전체 프로세스(#21) 중단/재시작 설계 필요**
전자결재 승인 대기 중 절차 중단 시나리오 발생.
→ 체크포인트 기록 + 재시작 지점 관리 설계 필요 (HR Day1과 동일한 이슈).
