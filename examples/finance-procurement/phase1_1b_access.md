## 체이닝 참여 분석

| 체이닝 유형 | 질의 예 | 접근 구간 시스템 |
|-----------|-------|--------------|
| 접근 단독 | 구매 요청 상태(#1), 예산 잔액(#2), 결재 상태(#3), 법인카드 내역(#5) | #9, #10, #12 |
| 접근+실행(확정) | 전체 프로세스(#21), 구매 요청(#15), 발주서(#16), 검수(#17), 지급(#18), 정산(#19), 결재(#20) | #9, #10, #11, #12, #14, #15, #16 |
| 접근+탐색+실행(유연) | 예산 초과 대응(#23) | #9, #10 |
| 접근+실행(유연) | 현황 리포트(#24), 벤더 납기 처리(#25), 벤더 선정(#22) | #9, #10, #11, #13, #14 |

> **핵심 패턴:** ERP(#9) + 전자결재(#10)는 거의 모든 실행 체이닝에 공통 참여. 회계(#11)는 지급 관련 체이닝에서 항상 추가.
>
> **"구매 컨텍스트 패키지" — Aggregator 패턴 적용:**
> ```
> Agent
>   └── Procurement Context Aggregator MCP (얇은 창구, 3~5개 tool)
>         ├── → ERP MCP — 구매 요청/발주/검수 현황
>         ├── → 전자결재 MCP — 결재 상태/결재선
>         └── → 회계 시스템 MCP — 전표/지급 상태
> ```
> `get_procurement_context(request_id)` 한 번 호출로 세 곳 상태를 반환. 에이전트는 Aggregator 하나만 알면 된다.

## 성숙도 메모

| 항목 | 초기 구현 | 성숙 시 |
|-----|---------|--------|
| ERP, 전자결재, 회계 시스템 | 직접 API 호출 | **Aggregator 패턴** MCP (구매 컨텍스트 패키지) — 최우선. Tool 3~5개. 세부 기능은 CLI |
| 법인카드, 벤더 DB, DMS | 직접 API 호출 | MCP 접근형 (실행 체이닝 시 승격). MCP는 얇게, CLI는 두텁게 |
| Gmail, Slack | 직접 API 호출 | 조회·발송 패턴 안정화 후 승격. 현재 우선순위 낮음 |
| 환율 API | 직접 API 호출 | 외화 결제 빈도 낮으면 CLI 수준 유지 |

> **전자결재 주의사항:** 전자결재(#10)는 독립 MCP 도구보다 파이프라인 내 **승인 게이트 단계**로 설계. 구매 요청 접수 MCP에서 "결재 상신 → 승인 대기" 단계로 내장되어야 함.
> Tool 컨텍스트 비용(1개당 550~1,400 토큰) 고려 — 도메인 전체 tool 수 50개 이하 설계.
