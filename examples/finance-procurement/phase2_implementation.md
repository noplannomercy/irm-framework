# Phase 2 — 구현 체크리스트

> 도메인: 재무/구매 승인
> 단계: Phase 2
> 상태: 완료
> 입력: phase1_1a ~ 1e 전체

---

## 개요

Phase 1의 배치 대상과 요구사항을 기반으로, 실제 구현 시 확인해야 할 의사결정 항목을 체크리스트로 정리한다.
"어떻게 결정하는가"는 IRM Framework v7 별첨 방법론을 참조한다.

---

## 구현 우선순위 요약

| 순서 | 대상 | 근거 |
|-----|-----|-----|
| 1 | 구매 컨텍스트 패키지 MCP (ERP+전자결재+회계) | 모든 실행(확정) 파이프라인의 선행 조건, ★★★ ×3 |
| 2 | RAG 구축 (finance-policy + vendor-data + budget + runbook) | 탐색 단독 질의 + 벤더 선정·예산 대응 실행(유연)의 선행 조건 |
| 3 | 실행(확정) 서브 파이프라인 MCP (구매 요청 접수, 결재 상신) | 구매 컨텍스트 패키지 MCP 완료 후 연결; 전자결재 승인 게이트 설계 포함 |
| 4 | 실행(확정) 서브 파이프라인 MCP (발주서 발행, 검수 처리) | Gmail·DMS MCP 완료 후 연결 |
| 5 | 실행(확정) 서브 파이프라인 MCP (지급 처리, 법인카드 정산) | 회계·법인카드 MCP 완료 후 연결; 지급 처리 승인 게이트 결정 포함 ⚠️ |
| 6 | 구매 전체 프로세스 파이프라인 MCP | #3,#4,#5 서브 파이프라인 전체 완료 후 오케스트레이션 |
| 7 | 실행(유연) Workflow (벤더 선정, 예산 초과 대응, 현황 리포트, 납기 이슈) | RAG + 구매 컨텍스트 패키지 MCP 완료 후 구성 |
| 8 | 나머지 시스템 MCP (Gmail, Slack, DMS, 법인카드, 벤더 DB) | 실행(확정) 파이프라인 연결 시점에 순차 구현 |

## 1. 탐색 구현 체크리스트

**입력: phase1_1a_search.md 1-A-2**

### 탐색(내부) — RAG 구축

- [ ] **1. 컬렉션 설계**
  - 갱신 주기와 검색 패턴 기준 컬렉션 분리 제안:
    - `finance-policy` — 구매 정책(#1), 법인카드 기준(#8), 감사 기준(#6), 회계 처리 기준(#7) / 정책 변경 시 갱신
    - `vendor-data` — 벤더 계약서(#2), 품목별 기준(#3), 벤더 평가이력(#5) / 계약·평가 시 갱신
    - `budget` — 예산 편성 문서(#4) / **분기별 갱신** (최신성 중요)
    - `runbook` — 절차 문서 #17~22 / 절차 변경 시 갱신
  - 벡터 저장소 제품 선정
  - → 별첨: RAG 설계 방법론 §1

- [ ] **2. 컬렉션별 메타데이터 스키마 확정**
  - 공통: `doc_id`, `source_url`, `updated_at`, `collection`
  - `finance-policy`: `policy_type`, `amount_range`, `approval_level`
  - `vendor-data`: `vendor_name`, `category`, `contract_status`
  - `budget`: `dept`, `quarter`, `budget_item`
  - `runbook`: `procedure_name`, `trigger_condition`
  - → 별첨: RAG 설계 방법론 §2

- [ ] **3. 문서 파싱 및 청킹**
  - Confluence 커넥터 연동 방식 결정 (구매 정책, 감사 기준, 절차 문서)
  - Google Drive PDF 파싱 방식 결정 (벤더 계약서 #2)
  - Notion 커넥터 연동 방식 결정 (벤더 평가이력 #5)
  - → 별첨: RAG 설계 방법론 §3

- [ ] **4. 임베딩 모델 선정**
  - 한국어 지원 여부 확인
  - → 별첨: RAG 설계 방법론 §4

- [ ] **5. 검색 전략 결정**
  - `vendor-data`: 벤더명 키워드+벡터 하이브리드 검색 권장
  - `budget`: `dept`, `quarter` 메타데이터 필터 필수 (최신 분기 우선)
  - `finance-policy`: `amount_range`, `approval_level` 필터 적용
  - **룩업(lookup) 패턴 최적화:** 이 도메인의 탐색은 "찾기"보다 "확인"이 중심 — 높은 정밀도 우선
  - → 별첨: RAG 설계 방법론 §5

- [ ] **6. 인덱싱 범위 결정**
  - `finance-policy`: 최신형 (현행 정책만)
  - `vendor-data`: 이력형 (계약 이력, 평가 이력 포함)
  - `budget`: 최신형 (현행 분기 예산) + 직전 분기 보관
  - `runbook`: 최신형 (현행 절차만)
  - → 별첨: RAG 설계 방법론 §7

- [ ] **7. 갱신 파이프라인 설계**
  - `budget`: **분기 시작 시 자동 갱신 필수** (예산 정보는 최신성이 가장 중요)
  - `vendor-data`: 계약 갱신·평가 완료 이벤트 트리거 연동
  - `finance-policy`: 정책 변경 시 수동 또는 Confluence 변경 이벤트 트리거
  - `runbook`: 절차 문서 변경 시 스케줄 배치
  - → 별첨: RAG 설계 방법론 §8

## 2. 접근 구현 체크리스트

**입력: phase1_1b_access.md 1-B-2**

- [ ] **1. MCP 대상 확정**
  - ★★★ 즉시 MCP로 시작: ERP(#9), 전자결재(#10), 회계(#11)
  - ★★ 운영 중 승격: 법인카드(#12), 벤더 DB(#13), Gmail(#14), DMS(#16)
  - ★ 승격 대기: Slack(#15), 환율 API(#25)
  - → 별첨: MCP 승격 설계 가이드 §1

- [ ] **2. MCP 서버 경계 결정**
  - **구매 컨텍스트 패키지 — Aggregator 패턴 적용:**
    - ERP + 전자결재 + 회계를 Aggregator MCP로 감쌈
    - 에이전트는 Aggregator 하나만 알면 됨. 안쪽 3개 시스템은 독립 MCP 유지
    - 근거: 구매 요청(#15), 지급 처리(#18), 전체 프로세스(#21), 현황 리포트(#24)에서 항상 세트로 호출
  - 법인카드, 벤더 DB, Gmail, DMS, Slack: 각각 개별 MCP 서버 (시스템 기준 분리 원칙)
  - **컨텍스트 비용 확인**: 도메인 전체 tool 수 50개 이하 유지
  - → 별첨: MCP 승격 설계 가이드 §2 (§2-0, §2-1, §2-3)

- [ ] **3. 도구 목록 및 그루핑 결정**
  - **MCP는 얇게, CLI는 두텁게** 원칙 적용:
    - Aggregator MCP tool 3~5개로 최소화
    - `get_procurement_context(request_id)` — ERP+전자결재+회계 통합 조회
    - `get_budget_status(dept, item)` — 예산 잔액+집행 현황 요약
    - `check_approval_status(doc_id)` — 결재 현황 확인
  - REST API 1:1 래핑 금지. 세부 기능은 erp-cli, approval-cli 등 CLI로
  - 도구 수 기준: 서버당 5~8개 이상적, 12개 초과 시 재검토
  - → 별첨: MCP 승격 설계 가이드 §2-2, §3

- [ ] **4. Transport 결정**
  - 재무팀+구매팀 공유 → HTTP (Streamable HTTP) 방향
  - 초기 개발·테스트는 stdio로 시작
  - → 별첨: MCP 승격 설계 가이드 §4

- [ ] **5. mcp-builder로 구현**
  - 위 1~4 결정사항 전달
  - → 별첨: MCP 승격 설계 가이드 §5

- [ ] **6. MCP 통합 검증**
  - 단독 조회 질의(#1~6)로 각 MCP 도구 동작 확인
  - 구매 전체 프로세스 진입 검증: Aggregator에서 정상 컨텍스트 반환 확인
  - → 별첨: MCP 승격 설계 가이드 §6

## 도메인별 주의 사항

1. **지급 처리 승인 게이트** ⚠️ — 실제 금액 이체 포함. 오발동 시 금전 피해. 자동 실행 전 현업 확인 필수. 금액 임계값 기준으로 자동/수동 분기 검토.
2. **ERP 선행 조건** — 7개 실행(확정) 파이프라인 대부분이 ERP 조회·생성으로 시작. ERP MCP 없이 어떤 파이프라인도 구현 불가. 구현 순서 1번.
3. **전자결재 승인 대기 설계** — 구매 요청(#15), 발주서(#16), 지급(#18)에서 전자결재 승인 대기 상태가 발생. 파이프라인이 승인 전에 중단되고 완료 후 재개하는 **비동기 처리 + 상태 추적** 설계 필요.
4. **구매 전체 프로세스 중단/재시작** — 4개 서브 파이프라인 중 하나가 실패해도 나머지를 이어서 실행하거나 재시작할 수 있도록 설계. 완료 상태를 단계별로 기록.
5. **budget 컬렉션 최신성** — 예산 편성 문서는 분기마다 갱신. RAG 인덱스가 오래된 예산 정보를 반환하면 잘못된 예산 확인으로 구매 요청 오류 발생. 분기 시작 시 자동 갱신 파이프라인 필수.
