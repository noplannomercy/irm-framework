## GraphRAG 승격 판단 (v8)

> v8 기준: 4개 조건 중 **2개 이상 충족 시** GraphRAG 승격을 검토한다.

| # | 승격 조건 | 충족 여부 | 근거 |
|---|---------|:-------:|------|
| 1 | 동음이의어/중의어가 많다 | **O** | "한도"(구매한도/예산한도/카드한도), "승인"(구매승인/결재승인/예산승인), "처리"(지급처리/검수처리/정산처리) — 같은 단어가 구매/회계/카드 도메인에서 다른 의미로 사용 |
| 2 | 관계/인과 기반 질의 패턴 발견 | **O** | #23 "예산 초과 → 대응" (예산잔액→구매요청→결재선 관계 추적), #22 "벤더 선정" (평가이력→계약조건→품목기준 관계 탐색) — 전체 탐색 질의 중 인과 패턴 약 25% |
| 3 | RAG 히트율 목표치 미달 | — | 운영 전이므로 판단 불가. 운영 후 측정 필요 |
| 4 | 동일 엔티티가 여러 도메인에 다른 맥락으로 등장 | **O** | "승인"이 구매승인(ERP)/결재승인(전자결재)/예산승인(예산시스템)으로 분산. 벡터 검색 시 세 맥락이 혼재될 가능성 높음 |

**충족: 3/4** → **승격 검토 대상**

**판단:** 조건 1, 2, 4가 이미 충족되어 있으므로, 운영 초기부터 GraphRAG를 적용해도 무방하다. 단, "RAG 먼저 → 필요 시 승격" 원칙을 따르는 것이 표준이다. 운영 중 RAG 히트율 측정(조건 3) 후 최종 승격 결정.

> **참조:** IRM Framework v8 — 성숙도 승격 기준 / GraphRAG 설계 방법론 v1

## 컬렉션 설계 제안

```
finance-policy     — 구매 정책, 법인카드 기준, 감사 기준, 회계 처리 기준
                     갱신: 정책 변경 시 / 검색 패턴: 정책 유형·금액대·결재선 기준
                     메타데이터: policy_type, amount_range, approval_level

vendor-data        — 벤더 계약서, 구매 품목별 기준, 벤더 평가 이력
                     갱신: 계약/평가 시 / 검색 패턴: 벤더명·품목·계약 상태 기준
                     메타데이터: vendor_name, category, contract_status

budget             — 예산 편성 문서
                     갱신: 분기별 / 검색 패턴: 부서·분기·항목 기준
                     메타데이터: dept, quarter, budget_item

runbook            — 절차 문서 #17~22 (지식 용도)
                     갱신: 절차 변경 시 / 검색 패턴: 절차명·트리거 조건 기준
                     메타데이터: procedure_name, trigger_condition
```

> **4개 컬렉션 기준:** 갱신 주기(정책/벤더계약/예산/절차)와 검색 패턴(정책조회/벤더비교/예산확인/절차참조)이 컬렉션마다 다르다.
>
> **HR 온보딩과 비교:** 탐색 성격이 "케이스 찾기"가 아니라 "기준 확인(룩업)" 중심. 컬렉션 수는 같지만 질의 패턴이 더 단순하고 명확하다. RAG 정밀도보다 **최신성**이 더 중요 (예산 편성 문서는 분기마다 갱신 필요).
