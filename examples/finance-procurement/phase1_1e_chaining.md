## 공통 접근 구간 — 구매 컨텍스트 패키지

실행을 포함하는 질의(#15~25) 중 **대부분이 ERP(#9)을 접근 구간에서 반드시 선행**한다.
ERP(#9) + 전자결재(#10) + 회계(#11)가 #15, #18, #21, #24에서 항상 세트로 호출.

```
[구매 컨텍스트 패키지] — MCP Aggregator 최우선 승격 대상
  ERP (#9)       — 구매 요청/발주/검수 현황, 예산 잔액
  전자결재 (#10)  — 결재 상태, 결재선, 승인 이력
  회계 (#11)     — 전표 현황, 지급 상태
```

이 세 시스템이 항상 세트로 조회되는 이유:
- 실행(확정) 파이프라인 진입 전 "현재 처리 상태 확인"이 필수
- 구매 전체 프로세스(#21)는 세 시스템의 현재 상태를 모두 알아야 다음 단계 결정 가능
- 현황 리포트(#24)도 세 곳 집계 필수

## 행위 유형별 컴포넌트 연결 구조

```
탐색(내부)
  └── RAG
        ├── finance-policy: 구매 정책, 법인카드 기준, 감사 기준, 회계 기준 (#7,#10,#11)
        ├── vendor-data: 벤더 계약서, 품목별 기준, 벤더 평가이력 (#8,#9,#22,#25)
        ├── budget: 예산 편성 문서 (#12,#23)
        └── runbook: 절차 문서 지식 (#23)

탐색(외부)
  └── 웹 검색
        ├── 국세청 세금계산서 법규 (#13)
        └── 공정거래위원회 규정 (#14)

접근
  └── 구매 컨텍스트 패키지 (MCP Aggregator형)
        ├── ERP (#9)
        ├── 전자결재 (#10)
        └── 회계 (#11)
  └── 실행 연계 시스템 (MCP 접근형)
        ├── 법인카드 (#12)
        ├── 벤더 DB (#13)
        ├── Gmail (#14)
        ├── Slack (#15)
        ├── DMS (#16)
        └── 환율 API (#25)

실행(확정)
  └── MCP 파이프라인형
        ├── 구매 요청 접수 (#15) → ERP + 전자결재 + Slack
        ├── 발주서 발행 (#16) → ERP + Gmail + DMS
        ├── 검수 처리 (#17) → ERP + DMS
        ├── 지급 처리 (#18) → ERP + 전자결재 + 회계 + DMS
        ├── 법인카드 정산 (#19) → 법인카드 + 회계 + DMS
        ├── 결재 상신 (#20) → 전자결재 + Slack
        └── 구매 전체 (#21) → #15→#16→#17→#18 순차 오케스트레이션

실행(유연)
  └── Workflow → MCP 에이전트형
        ├── 벤더 선정 (#22) → 벤더 DB + RAG[vendor-data]
        ├── 예산 초과 대응 (#23) → ERP + 전자결재 + RAG[budget,runbook]
        ├── 구매 현황 리포트 (#24) → 구매 패키지 Aggregator
        └── 벤더 납기 이슈 (#25) → ERP + 벤더 DB + Gmail
```

## 체이닝 흐름에서 드러나는 설계 시사점

**1. 구매 전체 프로세스 (#21)가 재무 도메인의 핵심 MCP**
4개 서브 파이프라인(#15→#16→#17→#18)을 순서대로 오케스트레이션.
→ 구매 전체 MCP가 서브 파이프라인 MCP들을 호출하는 계층 구조. 서브 파이프라인부터 구현 후 전체가 연결하는 순서.

**2. ERP가 모든 실행의 공통 첫 번째 단계**
접근 단독(#1~2) + 실행(확정) 5개(#15~18,#21) + 실행(유연) 3개(#23~25) = 10개+ 질의에서 ERP가 선행.
→ ERP MCP 없이 어떤 파이프라인도 구현 불가. 구현 1번.

**3. DMS가 실행(확정) 4개의 공통 마지막 단계**
발주서 발행(#16), 검수 처리(#17), 지급 처리(#18), 법인카드 정산(#19) 모두 DMS 증빙 등록으로 마무리.
→ HR 온보딩의 "Gmail이 4개의 공통 끝점"과 같은 패턴. DMS MCP를 공통 마지막 단계로 재사용 설계.

**4. 전자결재가 구조적 승인 게이트**
실행(확정) 4개(#15,#16,#18,#20)에서 전자결재가 파이프라인 중간에 삽입됨.
→ 전자결재는 독립 MCP가 아니라 파이프라인 내 "승인 대기 → 재개" 분기점으로 설계.
→ 승인 완료 콜백 또는 상태 폴링 메커니즘 필요.

**5. 실행(확정)이 7개로 세 도메인 최다 — MCP 파이프라인 집중 도메인**
SaaS CS: 실행(확정) 5개 vs 실행(유연) 4개
HR 온보딩: 실행(확정) 8개 vs 실행(유연) 5개
재무/구매: 실행(확정) 7개 vs 실행(유연) 4개
→ "정해진 순서의 반복 실행" 업무 특성. MCP 파이프라인형 설계가 핵심.
