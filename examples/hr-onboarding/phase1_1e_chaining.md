## 공통 접근 구간 — 입사자 컨텍스트 패키지

실행을 포함하는 질의(#13~25) 중 **대부분이 BambooHR(#9)을 접근 구간에서 반드시 선행**한다.

```
[입사자 컨텍스트 패키지] — MCP Aggregator 최우선 승격 대상
  BambooHR (#9)   — 입사자 정보, 계약, 부서, 직군
  Okta (#10)      — 계정·권한 상태
  AssetDB (#12)   — 장비 현황·배정 상태
  AD/LDAP (#16)   — 그룹·권한 상태
```

이 네 시스템이 항상 세트로 조회되는 이유:
- 실행(확정) 파이프라인 진입 전 "현재 상태 확인"이 필수
- Day1 전체 실행(#19)은 네 시스템의 현재 상태를 모두 알아야 진행 가능

## 행위 유형별 컴포넌트 연결 구조

```
탐색(내부)
  └── RAG
        ├── hr-policy: 온보딩 정책, 보안 정책, 복리후생, FAQ (#1~8)
        ├── dept-guide: 부서별 가이드, 조직도 (#2,#7)
        ├── training: 교육 자료 (#6,#17)
        └── runbook: 절차 문서 지식 (#17~22)

탐색(외부)
  └── 웹 검색
        ├── 고용노동부 근로기준 (#23)
        └── PIPC 개인정보 가이드라인 (#24)

접근
  └── 입사자 컨텍스트 패키지 (MCP Aggregator형)
        ├── BambooHR (#9)
        ├── Okta (#10)
        ├── AssetDB (#12)
        └── AD/LDAP (#16)
  └── 실행 연계 시스템 (MCP 접근형)
        ├── Gmail (#11)
        ├── 전자결재 (#14)
        ├── Jira SD (#15)
        └── Slack (#13)

실행(확정)
  └── MCP 파이프라인형
        ├── 계정 생성 (#13) → Okta + Gmail + Slack
        ├── 장비 배포 (#14) → AssetDB
        ├── 권한 배정 (#15) → AD/LDAP + Okta
        ├── 메일 발송 (#16) → Gmail
        ├── 교육 등록 (#17) → 전자결재
        ├── 서류 수집 (#18) → 전자결재 + Gmail
        ├── Day1 전체 (#19) → #13~#18 순차 오케스트레이션
        └── 퇴사 처리 (#25) → Okta + AD/LDAP + AssetDB + Gmail

  └── Workflow → MCP 에이전트형
        ├── 개발자 IT 세팅 (#20) → 입사자 패키지 + RAG

        └── 교육 미이수 리마인드 (#24) → BambooHR + 전자결재 + Gmail
```

## 체이닝 흐름에서 드러나는 설계 시사점

**1. Day1 파이프라인 (#19)이 온보딩 도메인의 핵심 MCP**
6개 서브 파이프라인(#13~#18)을 순서대로 오케스트레이션.
→ Day1 MCP가 서브 파이프라인 MCP들을 호출하는 계층 구조. 서브 파이프라인부터 구현 후 Day1이 연결하는 순서.

**2. BambooHR이 모든 실행의 공통 첫 번째 단계**
접근 단독(#1~5) + 실행(확정) 8개 + 실행(유연) 4개 = 17개 질의에서 BambooHR이 선행.
→ BambooHR MCP 없이 어떤 파이프라인도 구현 불가. 구현 1번.

**3. Gmail이 실행(확정) 4개의 공통 마지막 단계**
계정 생성(#13), 입사 안내 메일(#16), 서류 수집(#18), 퇴사 처리(#25) 모두 Gmail 발송으로 마무리.
→ Gmail MCP를 공통 마지막 단계로 재사용 설계 권장.

**4. 퇴사 처리 (#25)는 온보딩의 역방향 구조**
온보딩에서 생성한 계정/권한/장비를 역순으로 해제. 동일 시스템 세트.
→ 온보딩 MCP 서버 안에서 onboarding/offboarding 양방향 설계 가능.

**5. 실행(유연)이 적고 실행(확정)이 압도적으로 많다**
SaaS CS: 실행(확정) 5개 vs 실행(유연) 4개
HR 온보딩: 실행(확정) 8개 vs 실행(유연) 5개
→ 온보딩은 "정해진 절차의 반복 실행" 업무. MCP 파이프라인형 설계에 집중하면 된다.
