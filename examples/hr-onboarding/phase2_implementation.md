# Phase 2 — 구현 체크리스트

> 도메인: 신규 직원 온보딩 (HR)
> 단계: Phase 2
> 상태: 완료
> 입력: phase1_1a ~ 1e 전체

---

## 개요

Phase 1의 배치 대상과 요구사항을 기반으로, 실제 구현 시 확인해야 할 의사결정 항목을 체크리스트로 정리한다.
"어떻게 결정하는가"는 IRM Framework v7 별첨 방법론을 참조한다.

---

## 구현 우선순위 요약

| 순서 | 대상 | 근거 |
|-----|-----|-----|
| 1 | 입사자 컨텍스트 패키지 MCP (BambooHR+Okta+AssetDB+AD/LDAP) | 모든 실행(확정) 파이프라인의 선행 조건, ★★★ ×4 |
| 2 | RAG 구축 (hr-policy + dept-guide + training + runbook) | 탐색 단독 질의 + 실행(유연) 체이닝의 선행 조건 |
| 3 | 실행(확정) 서브 파이프라인 MCP (계정생성, 장비배포, 권한배정) | 입사자 컨텍스트 패키지 MCP 완료 후 연결 |
| 4 | 실행(확정) 서브 파이프라인 MCP (메일발송, 교육등록, 서류수집) | Gmail·전자결재 MCP 완료 후 연결 |
| 5 | Day1 온보딩 전체 파이프라인 MCP | #3,#4 서브 파이프라인 전체 완료 후 오케스트레이션 |
| 6 | 퇴사 처리 파이프라인 MCP | 온보딩 파이프라인 검증 완료 후 역방향 설계 |
| 7 | 실행(유연) Workflow (개발자 세팅, 현황 리포트, 미지급 처리, 리마인드) | RAG + 입사자 컨텍스트 패키지 MCP 완료 후 구성 |
| 8 | 나머지 시스템 MCP (Gmail, 전자결재, Jira SD, Slack) | 실행(확정) 파이프라인 연결 시점에 순차 구현 |

## 2. 접근 구현 체크리스트

**입력: phase1_1b_access.md 1-B-2**

- [ ] **1. MCP 대상 확정**

  - ★★ 운영 중 승격: Gmail(#11), 전자결재(#14)
  - ★ 승격 대기: Jira SD(#15), Slack(#13)
  - → 별첨: MCP 승격 설계 가이드 §1

- [ ] **2. MCP 서버 경계 결정**
  - **입사자 컨텍스트 패키지 — Aggregator 패턴 적용:**
    - BambooHR + Okta + AssetDB + AD/LDAP를 Aggregator MCP로 감쌈
    - 에이전트는 Aggregator 하나만 알면 됨. 안쪽 4개 시스템은 독립 MCP 유지
    - 근거: Day1 파이프라인 포함 모든 실행 체이닝에서 항상 세트로 선행
  - Gmail, 전자결재, Jira SD, Slack: 각각 개별 MCP 서버 (시스템 기준 분리 원칙)
  - **컨텍스트 비용 확인**: 도메인 전체 tool 수 50개 이하 유지
  - → 별첨: MCP 승격 설계 가이드 §2 (§2-0, §2-1, §2-3)

- [ ] **3. 도구 목록 및 그루핑 결정**
  - **MCP는 얇게, CLI는 두텁게** 원칙 적용:
    - Aggregator MCP tool 3~5개로 최소화
    - `get_onboarding_context(employee_id)` — BambooHR+Okta+AssetDB+AD/LDAP 통합 조회
    - `get_employee_status(employee_id)` — 계정·장비·권한 현재 상태 요약
    - `check_onboarding_progress(employee_id)` — 각 단계 완료 여부 확인
  - REST API 1:1 래핑 금지. 세부 기능은 bamboohr-cli, okta-cli 등 CLI로
  - 도구 수 기준: 서버당 5~8개 이상적, 12개 초과 시 재검토
  - → 별첨: MCP 승격 설계 가이드 §2-2, §3

- [ ] **4. Transport 결정**
  - HR팀+IT팀 공유 → HTTP (Streamable HTTP) 방향
  - 초기 개발·테스트는 stdio로 시작
  - → 별첨: MCP 승격 설계 가이드 §4

- [ ] **5. mcp-builder로 구현**
  - 위 1~4 결정사항 전달
  - → 별첨: MCP 승격 설계 가이드 §5

- [ ] **6. MCP 통합 검증**
  - 단독 조회 질의(#1~5)로 각 MCP 도구 동작 확인
  - Day1 파이프라인 진입 검증: Aggregator에서 정상 컨텍스트 반환 확인

## 도메인별 주의 사항

1. **퇴사 처리 승인 게이트** ⚠️ — 계정 비활성화는 즉시 접근 차단. 오발동 시 직원 피해. 자동 실행 전 현업 확인 필수.
2. **BambooHR 선행 조건** — 8개 실행(확정) 파이프라인 전체가 BambooHR 입사자 정보 확인으로 시작. BambooHR MCP 없이 어떤 파이프라인도 구현 불가. 구현 순서 1번.
3. **Day1 파이프라인 중단/재시작 설계** — 6개 서브 파이프라인 중 하나가 실패해도 나머지를 이어서 실행하거나 재시작할 수 있도록 설계. 완료 상태를 단계별로 기록.
4. **개인정보 처리** — 입사 서류 수집(#18)에 개인정보 포함. 전자결재 시스템의 보안 수준 확인 및 데이터 보존 기간 정책 반영 필요.
