## 체이닝 참여 분석

| 체이닝 유형 | 질의 예 | 접근 구간 시스템 |
|-----------|-------|--------------|
| 접근 단독 | 입사자 정보 조회(#1~5) | #9,#10,#12,#16 |
| 접근+실행(확정) | Day1 전체(#19), 계정생성(#13), 장비배포(#14), 권한배정(#15), 메일발송(#16), 교육등록(#17), 서류수집(#18), 퇴사처리(#25) | #9,#10,#11,#12,#13,#14,#16 |
| 접근+탐색+실행(유연) | 개발자 IT 세팅(#20) | #10,#16 |
| 접근+실행(유연) | 현황 리포트(#22), 미배정 처리(#23), 교육 리마인드(#24) | #9,#12,#14,#15 |

> **핵심 패턴:** BambooHR(#9) + Okta(#10) + AssetDB(#12) + AD/LDAP(#16)은 사실상 모든 실행 체이닝에 공통으로 포함된다.
>
> **"입사자 컨텍스트 패키지" — Aggregator 패턴 적용:**
> ```
> Agent
>   └── Onboarding Context Aggregator MCP (얇은 창구, 3~5개 tool)
>         ├── → BambooHR MCP — 입사자 정보
>         ├── → Okta MCP — 계정/권한 상태
>         ├── → AssetDB MCP — 장비 현황
>         └── → AD/LDAP MCP — 그룹·권한 상태
> ```
> `get_onboarding_context(employee_id)` 한 번 호출로 네 곳 상태를 반환. 에이전트는 Aggregator 하나만 알면 된다.

## 성숙도 메모

| 항목 | 초기 구현 | 성숙 시 |
|-----|---------|--------|
| BambooHR, Okta, AssetDB, AD/LDAP | 직접 API 호출 | **Aggregator 패턴** MCP (입사자 컨텍스트 패키지) — 최우선. Tool 3~5개. 세부 기능은 CLI |
| Gmail, 전자결재 | 직접 API 호출 | MCP 접근형 (실행 체이닝 시 자연스럽게 승격). MCP는 얇게, CLI는 두텁게 |
| Jira SD | 직접 API 호출 | 조회 빈도 확인 후 결정. 현황 리포트 빈도 높아지면 승격 |
| Slack | 직접 API 호출 | 승격 우선순위 낮음 — 단순 채널 초대·메시지 |

> **MCP 설계 핵심:** 이 도메인은 SaaS CS보다 실행(확정) 비중이 높다. Aggregator MCP가 접근 조회 뿐 아니라 Day1 파이프라인의 선행 조건으로도 작동한다. 구현 순서 1번 필수.
> Tool 컨텍스트 비용(1개당 550~1,400 토큰) 고려 — 도메인 전체 tool 수 50개 이하 설계.
