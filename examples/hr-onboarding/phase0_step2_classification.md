## 분류 기준

**Q1. 데이터 위치를 아는가?**
- 모른다 (사내 비정형 문서, 의미 기반 검색 필요) → 탐색(내부)
- 모른다 (사외 정보) → 탐색(외부)
- 안다 (시스템/API, 위치 확정) → 접근

**Q2. 처리 절차가 있는가?** → 있으면 Q3

**Q3. 이 절차는 결정적이고 재사용 가능한가?**
- Yes → 실행(확정)
- No → 실행(유연)

---

## 주목할 패턴

**1. 실행(확정) 비중이 높다**
10개 중 접근 시스템과 절차 문서가 섞여 있다. SaaS CS보다 표준화 범위가 넓다.
→ 온보딩 자체가 "정해진 절차의 반복 실행" 업무이기 때문. MCP 파이프라인형 후보가 많다.

**2. 실행(유연)이 2개뿐**
부서별 맞춤 온보딩(#2)과 개발자 IT 세팅(#20)만 해당.
→ 표준화 가능한 절차가 대부분이라 Workflow 필요성이 낮다. 단, 직군별 특수 케이스에서 판단 필요.

**3. 접근 시스템이 실행(확정)과 강하게 결합**
계정 생성(Okta), 장비 지급(AssetDB), 권한 배정(AD/LDAP) 모두 접근과 실행(확정)을 동시에 가진다.
→ SaaS CS의 "고객 컨텍스트 패키지"처럼 "입사자 온보딩 패키지"로 묶을 후보가 식별된다.
