## 설계 시사점

**1. Day1 파이프라인 (#19)이 모든 실행(확정)을 오케스트레이션한다**
#13~#18 여섯 절차가 #19의 서브 파이프라인으로 동작.
→ Day1 MCP가 각 서브 파이프라인 MCP를 순서대로 호출하는 계층 구조.

**2. BambooHR이 모든 실행(확정)의 첫 번째 단계**
8개 절차 전부 BambooHR 입사자 정보 확인으로 시작.
→ BambooHR MCP가 준비되지 않으면 어떤 실행(확정) 파이프라인도 구현 불가.
→ 구현 순서: 입사자 컨텍스트 패키지 MCP 먼저, 파이프라인 연결 그 다음.

**3. Gmail이 실행(확정) 4개의 공통 마지막 단계**
계정 생성(#13), 입사 안내(#16), 서류 수집(#18), 퇴사 처리(#25) 모두 Gmail 발송으로 마무리.
→ SaaS CS와 동일한 패턴. Gmail MCP를 공통 마지막 단계로 재사용 설계.

**4. 퇴사 처리 (#25)는 온보딩의 역방향 파이프라인**
계정 생성 → 계정 비활성화, 장비 배포 → 장비 반납. 동일 시스템 세트 사용.
→ 온보딩 MCP 서버에서 onboarding/offboarding 양방향 처리 가능하게 설계.

**5. 퇴사 처리 승인 게이트 여부 결정 필요**
계정 비활성화는 즉시 접근 차단 효과. 오발동 시 직원 피해.
→ 자동 실행 vs human-in-the-loop 확인 필요 → **현업 결정 필요** ⚠️
